## 背景：为什么微前端改造容易踩坑

微前端改造并非单纯的技术替换，而是**组织架构、开发流程、技术栈**三者的同步演进。ThoughtWorks 在 2016 年底的 Technology Radar 中首次提出微前端概念时，就强调其核心是"将前端应用视为多个特性的组合，每个特性由独立的全功能团队端到端负责"[^1]。

这意味着改造过程中最大的风险不是技术选型错误，而是**组织与架构不匹配**——如果团队仍然以集中式方式协作，微前端只会增加不必要的复杂度。

> "Just because you can does not mean that it's a wise idea to mix different technologies. Try to avoid Micro Frontends Anarchy."
>
> —— Michael Geers, [Micro Frontends](https://micro-frontends.org/)

## 改造前期的关键决策

### 1. 拆分边界的确定

**按业务域拆分**（推荐）而非按技术层拆分。每个微前端应该对应一个完整的业务能力，从数据库到用户界面。

| 拆分方式 | 示例 | 问题 |
|----------|------|------|
| **按业务域（推荐）** | 商品详情、购物车、订单管理、用户中心 | 团队端到端负责，部署独立 |
| 按技术层 | 头部导航、侧边栏、内容区、底部 | 任何业务变更都需要多个团队协调 |
| 按页面路由 | 每个路由一个微前端 | 粒度过细，通信成本剧增 |

**判断拆分是否合理的标准**：
- 该模块是否可以由一个团队独立开发、测试、部署？
- 该模块的业务变更是否通常不涉及其他模块？
- 该模块是否有独立的数据模型或 API？

### 2. 集成方式的选择

| 集成方式 | 适用场景 | 改造成本 | 独立部署 |
|----------|----------|----------|----------|
| **运行时 JS 集成** | 主流框架方案（qiankun/single-spa/wujie） | 中 | 是 |
| **运行时 iframe** | 第三方系统嵌入、强隔离需求 | 低 | 是 |
| **构建时集成** | 不推荐作为微前端方案 | 低 | 否（锁定发布） |
| **服务端 SSI/ESI** | 内容型网站、首屏性能敏感 | 中 | 是 |
| **Web Components** | 遵循 Web 标准、技术栈统一 | 中 | 是 |

**选型建议**：
- 遗留系统渐进式迁移 → **qiankun**（HTML Entry 接入成本最低）或 **wujie**（iframe 天然隔离）
- 从零搭建、需要最大灵活性 → **single-spa**（最基础抽象层）
- 纯 webpack 技术栈、需要细粒度模块共享 → **Module Federation**
- 对隔离性要求极高、交互简单 → **iframe**

### 3. 技术栈对齐策略

微前端允许技术栈无关，但**不意味着应该刻意追求技术栈多样化**[^2]。

**推荐策略**：
- **短期**：允许遗留系统保持原有技术栈，新模块采用统一技术栈
- **中期**：制定技术栈收敛路线图，逐步淘汰过时框架
- **长期**：保持 1-2 种主流框架，降低跨团队协作成本

## 改造过程中的常见问题与解决方案

### 样式冲突与隔离

**问题表现**：子应用样式覆盖主应用或其他子应用的样式，导致页面错乱。

**解决方案（按隔离强度排序）**：

| 方案 | 隔离强度 | 性能影响 | 适用场景 |
|------|----------|----------|----------|
| **BEM 命名规范** | 弱 | 无 | 团队纪律强、历史包袱轻 |
| **CSS Modules** | 中 | 编译时开销 | 构建时集成方案 |
| **Shadow DOM** | 强 | 轻微 | qiankun StrictStyleIsolation、wujie |
| **CSS-in-JS** | 强 | 运行时开销 | React 技术栈项目 |

**注意事项**：
- Shadow DOM 会导致全局组件（如弹窗、Tooltip）脱离隔离容器，需要特殊处理
- qiankun 的 `ScopedCSS` 方案标记为实验性，生产环境慎用
- 使用 Shadow DOM 时，第三方 UI 库（如 Ant Design）可能需要额外样式注入

### JavaScript 沙箱与全局变量污染

**问题表现**：多个子应用共享同一 window 对象，全局变量、事件监听器、定时器互相干扰。

**主流框架的沙箱方案**：

| 框架 | 沙箱机制 | 原理 | 限制 |
|------|----------|------|------|
| **qiankun** | ProxySandbox | 通过 Proxy 拦截全局变量读写 | 部分 API（如 `window.location`）无法完全代理 |
| **qiankun** | SnapshotSandbox | 定时快照对比，恢复变更 | 性能开销大，仅用于兼容旧浏览器 |
| **wujie** | iframe | 浏览器原生隔离 | 最安全，但带来路由、跨域等问题 |
| **single-spa** | 无内置 | 需自行实现 | 灵活性最高，但需要额外开发 |

**最佳实践**：
- 子应用代码中避免直接访问 `window` 上的全局变量
- 必须使用的全局状态应通过框架提供的通信机制传递
- 子应用卸载时清理所有副作用（事件监听器、定时器、WebSocket 连接）

### 路由与导航管理

**问题表现**：主应用与子应用路由冲突、浏览器前进后退行为异常、刷新后页面状态丢失。

**解决方案**：

1. **路由前缀隔离**：每个子应用使用独立的路由前缀
   ```js
   // qiankun 配置示例
   registerMicroApps([
     { name: 'app1', activeRule: '/app1' },
     { name: 'app2', activeRule: '/app2' },
   ]);
   ```

2. **子应用路由基址设置**：子应用需配置 `publicPath` 或 `base`
   ```js
   // Vue Router 示例
   const router = new VueRouter({
     base: window.__POWERED_BY_QIANKUN__ ? '/app1' : '/',
     mode: 'history',
   });
   ```

3. **单例路由实例**：子应用导出路由实例而非创建新实例，避免多次 mount 时重复创建

### 依赖共享与版本冲突

**问题表现**：多个子应用加载相同依赖的不同版本，导致产物体积膨胀、运行时行为不一致。

**解决方案**：

| 方案 | 适用框架 | 说明 |
|------|----------|------|
| **Import Maps** | single-spa（推荐） | 浏览器原生模块解析，统一版本管理 |
| **Module Federation Shared** | webpack 5 | 构建时共享，运行时去重 |
| **External 配置** | 通用 | 将公共依赖标记为 external，由主应用提供 |
| **统一版本锁定** | 通用 | 通过 monorepo 或依赖管理工具锁定版本 |

**关键原则**[^3]：
- 大型框架依赖（React、Vue、Angular）**必须共享**，只加载一次
- 小型工具库可以接受重复加载，便于独立升级
- 共享依赖升级需要所有使用方同步更新，需谨慎评估

### 跨应用通信

**问题表现**：子应用之间需要传递状态或触发行为，直接引用导致耦合。

**通信方式对比**：

| 方式 | 耦合度 | 适用场景 | 示例 |
|------|--------|----------|------|
| **URL 参数** | 最低 | 页面级状态传递 | `?productId=123` |
| **CustomEvent** | 低 | 跨应用事件通知 | `window.dispatchEvent(new CustomEvent('app:basket:changed'))` |
| **Props 传递** | 中 | 主应用向子应用下发数据 | qiankun 的 `props` |
| **全局状态** | 高 | 多应用共享复杂状态 | qiankun `initGlobalState`、Redux |

**最佳实践**[^3]：
- 优先使用 URL 参数传递状态（最解耦、可分享、支持刷新）
- 如果两个应用频繁传递 UI 状态，考虑合并为一个微前端
- 避免使用全局 Redux/MobX store，这会破坏微前端的独立部署能力

### 性能优化

**常见问题**：
- 首屏加载多个子应用的 JS 资源，导致白屏时间过长
- 沙箱和样式隔离带来运行时开销
- 子应用切换时重复加载资源

**优化策略**：

| 策略 | 说明 | 实现方式 |
|------|------|----------|
| **按需加载** | 仅加载当前路由需要的子应用 | single-spa `activeWhen`、qiankun `activeRule` |
| **预加载** | 浏览器空闲时预加载可能访问的子应用 | qiankun `prefetch: 'all'` |
| **子应用保活** | 缓存子应用状态，切换时不重新渲染 | wujie 原生支持、qiankun `keepAlive` |
| **Skeleton Screen** | 骨架屏占位，避免布局抖动 | 服务端渲染骨架 + 客户端填充 |
| **资源缓存** | 利用 HTTP 缓存策略减少重复请求 | 文件名 hash + 长期缓存 |

## 后期维护实践

### 版本管理与升级策略

**公共依赖升级**：
- 建立共享依赖仓库（如 single-spa 推荐的 [shared-dependencies repo](https://github.com/polyglot-microfrontends/shared-dependencies)）
- 通过 CI 流程统一更新 import map 中的依赖版本
- 升级前在所有子应用中验证兼容性

**框架版本升级**：
- 微前端架构的优势之一是**可以逐个升级子应用**
- 制定升级路线图：先升级非核心业务，再升级核心业务
- 保持主应用框架版本相对稳定，避免频繁变更影响所有子应用

### 本地开发体验

**问题**：微前端改造后，本地开发需要启动多个服务，开发体验下降。

**推荐方案**[^3]：
- **import-map-overrides**：通过浏览器插件将某个子应用的地址指向 localhost，其他子应用使用线上版本
- **standalone 模式**：子应用支持独立运行，不依赖主应用
- **Mock 服务**：主应用提供 Mock 数据，子应用无需连接真实后端

**典型工作流**：
```
1. 启动主应用的线上/测试环境
2. 本地启动需要开发的子应用
3. 通过 import-map-overrides 将子应用指向 localhost
4. 开发完成后部署子应用，更新 import map
```

### CI/CD 流水线设计

**核心原则**：每个微前端应该有**独立的 CI/CD 流水线**[^3]。

**部署两步走**：
1. 将构建产物上传到 CDN（如 AWS S3 + CloudFront）
2. 更新 import map 指向新版本文件

**并发安全**：
- 多个团队同时部署时，import map 更新可能产生竞态条件
- 使用 [import-map-deployer](https://github.com/single-spa/import-map-deployer) 等工具保证并发安全
- 或采用文件名 hash 策略，通过 302 重定向指向最新版本，避免直接修改 import map

### 监控与错误追踪

**监控维度**：
- **应用级监控**：每个子应用独立上报错误和性能指标
- **集成级监控**：主应用监控子应用加载失败、路由异常等集成问题
- **用户体验监控**：首屏时间、子应用切换耗时、内存使用

**错误隔离**：
- 子应用的 JavaScript 错误不应影响主应用和其他子应用
- 使用 `window.onerror` 或 `ErrorBoundary` 捕获并上报子应用错误
- qiankun 提供 `addErrorHandler` API 统一处理子应用错误

### 测试策略

| 测试层级 | 负责方 | 测试内容 |
|----------|--------|----------|
| **单元测试** | 各子应用团队 | 组件逻辑、工具函数 |
| **集成测试** | 各子应用团队 | 子应用内部模块交互 |
| **契约测试** | 跨团队协作 | 主应用与子应用的接口契约 |
| **E2E 测试** | 主应用团队 | 页面组装正确性、路由跳转 |

**契约测试**（Contract Testing）是微前端架构中保障跨团队协作的关键手段，详细内容可参考 [[契约测试（Contract Testing）]]。

### 文档与规范维护

**必须维护的文档**：
- **接入规范**：子应用接入主应用的步骤、配置要求、生命周期说明
- **通信协议**：跨应用通信的事件名称、数据格式、Props 定义
- **样式规范**：CSS 命名约定、主题变量、全局样式使用限制
- **发布流程**：子应用发布步骤、import map 更新流程、回滚方案

**规范治理**：
- 建立架构评审机制，新子应用接入前进行方案评审
- 定期回顾规范执行情况，收集团队反馈
- 使用脚手架工具固化最佳实践，降低接入成本

## 常见误区

| 误区 | 正确认知 |
|------|----------|
| "微前端就是拆得越细越好" | 拆分过细会增加通信和维护成本，应按业务域合理拆分 |
| "每个子应用都应该用不同框架" | 技术栈多样化会增加协作成本，应保持适度统一 |
| "微前端不需要全局状态管理" | 不是不需要，而是应避免单一全局 store，改用局部状态 + 事件通信 |
| "iframe 是最简单的方案" | iframe 接入简单，但路由、弹窗、跨域等问题后续处理成本高 |
| "微前端能解决所有性能问题" | 微前端本身不提升性能，合理拆分和优化才能改善体验 |
| "改造后团队就可以完全独立" | 仍需要协调接口契约、样式规范、公共依赖升级等事项 |
