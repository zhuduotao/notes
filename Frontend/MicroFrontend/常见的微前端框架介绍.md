## 什么是微前端

微前端（Micro Frontends）是一种将前端应用拆分为多个**可独立开发、独立部署**的小型应用的架构风格。这一概念最早由 ThoughtWorks 在 2016 年底的 [Technology Radar](https://www.thoughtworks.com/radar/techniques/micro-frontends) 中提出，其核心思想是将微服务的理念延伸到前端领域。

> "An architectural style where independently deliverable frontend applications are composed into a greater whole"
>
> —— Cam Jackson, [Micro Frontends](https://martinfowler.com/articles/micro-frontends.html), Martin Fowler Blog, 2019

微前端将网站或 Web 应用视为**多个特性的组合**，每个特性由**独立的团队**负责，团队跨职能且端到端地开发其特性（从数据库到用户界面）。

## 为什么需要微前端

### 解决前端单体困境

随着业务增长，前端项目往往演变为难以维护的**前端单体（Frontend Monolith）**：
- 代码库庞大，构建时间越来越长
- 多团队协作时代码冲突频繁
- 技术栈升级困难，"牵一发而动全身"
- 新功能上线需要等待整个应用发布

### 核心优势

| 优势 | 说明 |
|------|------|
| **渐进式升级** | 可逐模块替换旧技术栈，避免全量重写风险 |
| **代码解耦** | 每个微前端代码库更小、更清晰，减少意外耦合 |
| **独立部署** | 每个模块有自己的 CI/CD 流水线，部署互不影响 |
| **团队自治** | 团队可自主选择技术栈，端到端负责特定业务域 |
| **技术栈无关** | 允许同一页面混合使用 React、Vue、Angular 等框架 |

### 适用场景

- 大型应用由多个业务域组成，适合不同团队独立维护
- 需要对遗留系统进行渐进式重构
- 多团队并行开发，需要减少代码冲突和协调成本
- 需要在同一页面集成不同技术栈的模块

## 微前端的集成方式

微前端有多种集成方案，各有优劣：

### 1. 服务端模板组合（Server-side Template Composition）

通过 Nginx SSI（Server Side Includes）或 ESI（Edge Side Includes）在服务端将多个 HTML 片段组合成完整页面。

- **优点**：实现简单，首屏性能好，天然支持 JavaScript 禁用场景
- **缺点**：服务端耦合，动态交互能力有限
- **适用**：内容型网站、对首屏性能要求高的场景

### 2. 构建时集成（Build-time Integration）

将每个微前端打包为 npm 包，由容器应用统一引入并构建。

- **优点**：可去重公共依赖，产物体积小
- **缺点**：**锁定发布**——任何模块变更都需要重新构建发布整个应用，违背微前端独立部署初衷
- **结论**：不推荐作为微前端方案 [^1]

### 3. 运行时集成 — iframe

通过 iframe 将子应用嵌入容器页面。

- **优点**：天然隔离（CSS、JS、全局变量零干扰），实现成本极低
- **缺点**：路由/历史记录管理复杂、弹窗/遮罩层受限、响应式适配困难、用户体验割裂
- **适用**：对隔离性要求极高、交互简单的场景（如第三方后台嵌入）

### 4. 运行时集成 — JavaScript

每个微前端通过 `<script>` 标签加载，暴露全局入口函数，容器应用根据路由决定何时何地渲染。

- **优点**：灵活性最高，可深度定制通信和渲染逻辑
- **缺点**：需要自行处理沙箱隔离、样式冲突、依赖共享等问题
- **适用**：需要高度定制化的场景，也是当前主流框架的基础方案

### 5. 运行时集成 — Web Components

每个微前端注册为 Custom Element，容器应用通过创建自定义元素实例来渲染。

- **优点**：基于 Web 标准，Shadow DOM 提供天然样式隔离
- **缺点**：框架兼容性需注意（如 React 对 Custom Events 的处理）
- **适用**：希望遵循 Web 标准、技术栈相对统一的团队

## 主流微前端框架对比

### single-spa

| 项目 | 信息 |
|------|------|
| **官方仓库** | [single-spa/single-spa](https://github.com/single-spa/single-spa) |
| **当前版本** | 6.x |
| **核心原理** | 路由劫持 + 应用生命周期管理 |
| **沙箱机制** | 无内置沙箱，需自行实现 |
| **样式隔离** | 无内置方案，需自行处理 |
| **技术栈** | 框架无关 |

single-spa 是最早的微前端框架之一，由 Canopy 公司开源。它将现代框架的组件生命周期抽象为**应用级别的生命周期**（bootstrap、mount、unmount）。

**核心概念**：
- **Root Config**：根配置，负责注册应用、渲染 HTML 页面
- **Application**：微应用，需实现 bootstrap/mount/unmount 生命周期
- **Parcel**：可复用的组件单元，可在任何框架中使用

**典型用法**：
```js
import { registerApplication, start } from 'single-spa';

registerApplication({
  name: 'app1',
  app: () => import('app1/main.js'),
  activeWhen: ['/app1'],
});

start();
```

**优点**：
- 成熟稳定，社区生态完善
- 支持任意 JavaScript 框架共存
- 提供 `create-single-spa` 脚手架快速上手
- 支持 import maps 实现共享依赖

**缺点**：
- 不提供 JS 沙箱和 CSS 隔离，需自行实现
- 上手门槛较高，需要理解底层架构
- 子应用需要改造以适配生命周期

**官方文档**：[single-spa.js.org](https://single-spa.js.org/docs/getting-started-overview/)

### qiankun（乾坤）

| 项目 | 信息 |
|------|------|
| **官方仓库** | [umijs/qiankun](https://github.com/umijs/qiankun) |
| **当前版本** | 2.x |
| **核心原理** | 基于 single-spa 封装 + HTML Entry + 沙箱 |
| **沙箱机制** | SnapshotSandbox（兼容模式）、ProxySandbox（现代浏览器） |
| **样式隔离** | StrictStyleIsolation（Shadow DOM）、ScopedCSS（实验性） |
| **技术栈** | 框架无关 |

qiankun 由蚂蚁集团开源，自称"最完整的微前端解决方案"。它在 single-spa 基础上提供了开箱即用的能力。

**核心特性**：
- **HTML Entry**：直接通过 URL 加载子应用，类似 iframe 的使用体验
- **JS 沙箱**：通过 Proxy 或快照隔离子应用的全局变量
- **样式隔离**：支持 Shadow DOM 严格隔离或 Scoped CSS
- **预加载**：空闲时预加载子应用资源
- **应用通信**：通过 `initGlobalState` 提供全局状态管理

**典型用法**：
```js
import { registerMicroApps, start } from 'qiankun';

registerMicroApps([
  {
    name: 'reactApp',
    entry: '//localhost:7100',
    container: '#container',
    activeRule: '/react',
  },
]);

start({ prefetch: 'all' });
```

**优点**：
- 基于 single-spa，生态继承良好
- HTML Entry 接入成本低，子应用几乎无需改造
- 内置沙箱和样式隔离方案
- 生产经验丰富（蚂蚁集团内部大量使用）

**缺点**：
- Shadow DOM 隔离可能导致弹窗等组件样式异常
- Proxy 沙箱在部分场景下存在兼容性问题
- 性能开销（沙箱、样式隔离）需关注
- 2.x 版本维护节奏放缓

**官方文档**：[qiankun.umijs.org](https://qiankun.umijs.org/)

### wujie（无界）

| 项目 | 信息 |
|------|------|
| **官方仓库** | [Tencent/wujie](https://github.com/Tencent/wujie) |
| **核心原理** | Web Components + iframe 沙箱 |
| **沙箱机制** | iframe（天然 JS 隔离） |
| **样式隔离** | Web Components Shadow DOM |
| **技术栈** | 框架无关 |

wujie 由腾讯团队开源，采用 **Web Components + iframe** 的独特架构，兼顾隔离性和性能。

**核心特性**：
- **原生隔离**：JS 在 iframe 中运行，CSS 通过 Shadow DOM 隔离
- **子应用保活**：支持子应用状态缓存，切换时不重新渲染
- **子应用嵌套**：支持微前端多层嵌套
- **多应用激活**：同一页面可同时激活多个子应用
- **vite 支持**：原生支持 vite 构建的子应用
- **去中心化通信**：基于自定义事件的通信机制

**典型用法**：
```js
import { startApp } from 'wujie';

startApp({
  name: 'sub-app-id',
  url: 'http://localhost:3000',
  el: '#container',
  sync: true,
});
```

**优点**：
- iframe 提供最强隔离，几乎无 JS 冲突风险
- 子应用接入成本极低，几乎零改造
- 子应用保活体验好，切换无重新加载
- 支持 vite 框架

**缺点**：
- iframe 带来的路由、弹窗、跨域等问题需额外处理
- 相对较新，社区生态不如 single-spa/qiankun 成熟
- 多 iframe 场景下内存占用需关注

**官方文档**：[wujie-micro.github.io/doc](https://wujie-micro.github.io/doc/)

### icestark

| 项目 | 信息 |
|------|------|
| **官方仓库** | [ice-lab/icestark](https://github.com/ice-lab/icestark) |
| **核心原理** | 路由劫持 + JS Entry / HTML Entry |
| **沙箱机制** | Proxy API（JavaScript 沙箱） |
| **样式隔离** | 无内置强隔离方案 |
| **技术栈** | 主/子应用框架无关 |

icestark 由阿里巴巴 ice 团队开源，面向**大型应用**的微前端解决方案。

**核心概念**：
- **主应用（Main-application）**：负责子应用注册、加载、渲染和布局
- **子应用（Sub-application）**：负责自身业务内容展示

**典型用法**：
```js
import { registerMicroApps, start } from '@ice/stark';

registerMicroApps([
  {
    name: 'app1',
    activePath: ['/app1'],
    container: document.getElementById('container'),
    url: ['//cdn.example.com/app1/index.js'],
  },
]);

start();
```

**子应用接入方式**：
1. **注册生命周期**：通过 `@ice/stark-app` 提供的 `registerAppEnter`/`registerAppLeave`
2. **导出 UMD 生命周期**：打包为 UMD 模块，暴露 `mount`/`unmount` 函数

**优点**：
- 提供 React 和 Vue 的脚手架，快速搭建
- 支持 JS Entry 和 HTML Entry 多种接入方式
- 兼容 single-spa 的子应用生命周期
- 阿里内部大规模使用验证

**缺点**：
- 社区活跃度相对较低
- 样式隔离方案不如 qiankun 完善
- 文档以中文为主，国际化程度有限

**官方文档**：[micro-frontends.ice.work](https://micro-frontends.ice.work/)

### webpack Module Federation

| 项目 | 信息 |
|------|------|
| **官方文档** | [webpack Module Federation](https://webpack.js.org/concepts/module-federation/) |
| **引入版本** | webpack 5 |
| **核心原理** | 模块级别的运行时共享 |
| **沙箱机制** | 无 |
| **样式隔离** | 无 |
| **技术栈** | 仅 webpack 项目 |

Module Federation 是 webpack 5 引入的**模块联邦**特性，允许多个独立的构建产物在运行时形成一个完整应用。

**核心概念**：
- **Container**：每个构建既是容器（暴露模块），也消费其他容器的模块
- **Remote**：被其他容器引用的远程模块
- **Shared**：共享模块（如 React），避免重复加载

**典型配置**：
```js
// 宿主应用 webpack.config.js
new ModuleFederationPlugin({
  name: 'host',
  remotes: {
    app1: 'app1@http://localhost:3001/remoteEntry.js',
  },
  shared: ['react', 'react-dom'],
});

// 子应用 webpack.config.js
new ModuleFederationPlugin({
  name: 'app1',
  filename: 'remoteEntry.js',
  exposes: {
    './Button': './src/Button',
  },
  shared: ['react', 'react-dom'],
});
```

**优点**：
- 构建工具原生支持，无需额外运行时框架
- 模块级别共享，粒度更细，依赖去重效果好
- 支持动态远程容器（Dynamic Remotes），可运行时切换版本
- 适合组件库共享、多团队共建大型应用的场景

**缺点**：
- 仅限 webpack 5 生态，非 webpack 项目无法直接使用
- 不提供 JS 沙箱和样式隔离，需自行处理
- 配置复杂度较高，需理解 shared/exposes/remotes 等概念
- 版本冲突处理需谨慎（`requiredVersion` 机制）

**官方文档**：[webpack Module Federation](https://webpack.js.org/concepts/module-federation/)

| 特性 | single-spa | qiankun | wujie | icestark | Module Federation |

| **接入成本** | 中（需改生命周期） | 低（HTML Entry） | 极低 | 低 | 中（需改 webpack 配置） |

| **独立部署** | 是 | 是 | 是 | 是 | 是 |
| **子应用保活** | 需自行实现 | 支持 | 原生支持 | 支持 | 不适用 |
| **vite 支持** | 需适配 | 需适配 | 原生支持 | 需适配 | 不适用 |
| **社区成熟度** | 高 | 高 | 中 | 中 | 高 |

## 框架选型建议

| 场景 | 推荐方案 | 依据 |
|------|:--------:|------|
| 需要最强隔离，子应用几乎不改造 | **wujie** | iframe + Web Components 提供原生隔离 |
| 已有 single-spa 基础，需要开箱即用的沙箱和样式隔离 | **qiankun** | 生态成熟，功能完整 |
| 纯 webpack 技术栈，需要细粒度模块共享 | **Module Federation** | 构建工具原生支持，依赖去重效果好 |
| 需要从零搭建，希望有脚手架快速上手 | **icestark** | 提供 React/Vue 脚手架 |
| 需要最大灵活性和控制权 | **single-spa** | 最基础的抽象层，可自由扩展 |
| 遗留系统渐进式迁移 | **qiankun / wujie** | HTML Entry / iframe 方案接入成本最低 |

## 常见问题与注意事项

### 样式冲突

微前端最常见问题之一是样式污染。解决方案包括：
- **BEM 命名规范**：通过约定减少冲突，依赖团队纪律
- **CSS Modules / CSS-in-JS**：编译时或运行时生成唯一类名
- **Shadow DOM**：浏览器原生样式隔离，但可能影响弹窗等全局组件
- **团队前缀约定**：如 `[team]-[component]` 命名规范

### 全局变量污染

多个子应用共享同一页面，全局变量冲突不可避免：
- 使用沙箱机制隔离子应用运行环境
- 避免使用全局变量，改用模块导入导出
- 如必须使用全局变量，采用命名空间前缀区分

### 依赖重复加载

不同子应用可能引入相同依赖的不同版本，导致：
- 产物体积增大，加载性能下降
- 运行时行为不一致

**解决方案**：
- 使用 import maps（single-spa 推荐）或 Module Federation shared 机制
- 统一公共依赖版本
- 将常用库（React、Vue 等）提取为外部依赖

### 跨应用通信

微前端之间应**尽量减少直接通信**，避免重新引入耦合：
- **路由通信**：通过 URL 参数传递状态（推荐，最解耦）
- **自定义事件**：`CustomEvent` 实现发布/订阅模式
- **全局状态**：qiankun 的 `initGlobalState` 或 Redux 等状态管理库
- **Props 传递**：容器应用向子应用下发数据

### 性能考量

- **首屏加载**：多个子应用的 JS 资源可能增加首屏时间，建议按需加载和预加载结合
- **沙箱开销**：Proxy 沙箱和 Shadow DOM 会带来一定性能损耗
- **内存泄漏**：子应用卸载时需清理事件监听器、定时器等资源

### 测试策略

- 每个微前端应有独立的单元测试和组件测试
- 容器应用编写少量 E2E 测试验证页面组装正确性
- 跨应用交互可使用消费者驱动契约（Consumer-Driven Contracts）测试
