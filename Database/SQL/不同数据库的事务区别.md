# 不同数据库的事务区别

---

## 写偏斜（Write Skew）深度解析

### 是什么

**写偏斜（Write Skew）** 是一种在并发事务中出现的异常现象，属于**隔离级别**问题。它通常发生在 **快照隔离（Snapshot Isolation, SI）** 或 **可重复读（REPEATABLE READ）** 级别下。

当两个事务同时读取了**重叠的数据集**，然后分别更新了**各自读取的不同部分**，并且提交后破坏了数据库的**整体一致性约束**时，就发生了写偏斜。

简单来说：每个事务单独看都是合法的，但两个事务一起执行后，结果却是非法的。

### 经典案例：医生值班

假设医院有一条规则：**任何时刻，至少必须有一名医生在值班。**
数据库中有两条记录：`Doctor A` (值班中) 和 `Doctor B` (值班中)。

#### 场景复现：
1.  **事务 1 (T1)**：医生 A 想下班。
    *   **读取**：T1 查询当前值班医生，看到 A 和 B 都在值班。
    *   **判断**：既然 B 还在，我下班不会违反规则。
    *   **更新**：T1 将 A 的状态改为“不在”。
2.  **事务 2 (T2)**：医生 B 也想下班（几乎同时发生）。
    *   **读取**：T2 查询当前值班医生（基于快照），看到 A 和 B 都在值班（因为 T1 还没提交，或者在 SI 下 T2 看不到 T1 的修改）。
    *   **判断**：既然 A 还在，我下班不会违反规则。
    *   **更新**：T2 将 B 的状态改为“不在”。
3.  **提交**：
    *   T1 提交成功。
    *   T2 提交成功。

**结果**：A 和 B 都不在值班了，**违反了“至少一名医生在值班”的约束**。这就是写偏斜。

### 为什么叫“偏斜” (Skew)？

因为两个事务基于**不同的快照视图**做出了决策。
*   T1 看到的世界：{A: 在, B: 在} -> 决定 A 离开。
*   T2 看到的世界：{A: 在, B: 在} -> 决定 B 离开。
*   真实的世界：{A: 不在, B: 不在}。

两个事务的视角发生了“错位”或“偏斜”，导致它们忽略了彼此的存在。

### 与其他异常的区别

| 异常类型 | 现象描述 | 例子 |
| :--- | :--- | :--- |
| **脏读** | 读到别人没提交的数据 | 读到别人正在修改但未确认的余额。 |
| **不可重复读** | 同一个事务内，两次读同一行不一样 | 第一次读余额 100，第二次读变成 200。 |
| **幻读** | 同一个事务内，两次范围查询行数不一样 | 第一次查 3 人，第二次查变成 4 人。 |
| **写偏斜** | **两个事务更新不同行，但破坏了关联约束** | 两个医生都以为对方在，结果都下班了。 |

> **注意**：写偏斜比幻读更难理解，因为它不仅仅涉及行数的变化，还涉及**业务逻辑约束**的破坏。

## 不同数据库的事务处理区别

### 1. 隔离级别实现差异

不同数据库对 SQL 标准隔离级别的实现并不完全一致，主要体现在底层并发控制机制上。

| 数据库 | 支持的隔离级别 | 默认级别 | 核心实现机制 | 写偏斜风险 |
| :--- | :--- | :--- | :--- | :--- |
| **MySQL (InnoDB)** | `READ UNCOMMITTED`, `READ COMMITTED`, `REPEATABLE READ`, `SERIALIZABLE` | `REPEATABLE READ` | MVCC + Next-Key 锁 | **低**（Next-Key 锁在多数场景下意外避免了写偏斜，但纯快照读仍可能发生） |
| **PostgreSQL** | `READ COMMITTED`, `REPEATABLE READ`, `SERIALIZABLE` | `READ COMMITTED` | MVCC + SSI | **无**（SSI 模式可检测并回滚写偏斜） |
| **TiDB** | `READ COMMITTED`, `REPEATABLE READ` | `REPEATABLE READ` | MVCC + 快照隔离 (SI) | **有**（SI 允许写偏斜） |
| **MongoDB** | `Snapshot Isolation` | `Snapshot Isolation` | MVCC + 快照 | **有**（快照隔离允许写偏斜） |
| **SurrealDB** | `SERIALIZABLE` | `SERIALIZABLE` | MVCC + 多版本快照 | **无**（串行化保证） |
| **DynamoDB** | `SERIALIZABLE` | `SERIALIZABLE` | 2PC 变体 + 锁 | **无**（串行化保证） |

### 2. 关键区别详解

#### MySQL (InnoDB) vs PostgreSQL
*   **MySQL** 的 `REPEATABLE READ` 实际上比 SQL 标准更强，因为它使用了 Next-Key 锁（记录锁 + 间隙锁），这在很大程度上防止了幻读，也在很多场景下防止了写偏斜（因为写操作会加锁）。但在某些纯快照读（不加锁）的复杂更新场景下，仍可能存在理论上的写偏斜风险。
*   **PostgreSQL** 的 `REPEATABLE READ` 是标准的快照隔离（SI），**明确允许写偏斜**。但 PG 提供了 `SERIALIZABLE` 级别，使用了 **Serializable Snapshot Isolation (SSI)** 技术，能够自动检测读写依赖冲突，如果发现可能导致写偏斜，会主动回滚其中一个事务（抛出 `serialization failure`）。

#### TiDB 的特殊性
*   TiDB 的 `REPEATABLE READ` 实现的是 **Snapshot Isolation (SI)**。
*   它**禁止严格幻读**，但**允许写偏斜**。
*   在 TiDB 中，如果业务逻辑强依赖跨行约束（如医生值班案例），必须显式使用 `SELECT ... FOR UPDATE` 加锁，或者在应用层处理冲突。

#### NoSQL 数据库的事务
*   **MongoDB**：多文档事务提供快照隔离。这意味着在并发更新不同文档时，如果存在业务关联约束，开发者需要自己处理写偏斜问题（通常通过乐观锁版本号或应用层重试）。
*   **DynamoDB**：通过 `TransactWriteItems` 等 API 提供**串行化**隔离。底层使用两阶段提交（2PC）的变体，对涉及的数据项加锁，因此**天然杜绝写偏斜**，但代价是并发吞吐量受限。

## 如何避免写偏斜？

如果你在使用支持 SI 的数据库（如 TiDB, PostgreSQL 的 RR 级别, MongoDB），可以通过以下方式避免：

### 1. 显式加锁（SELECT ... FOR UPDATE）
在读取数据时加上排他锁，将快照读转换为当前读。
```sql
-- 读取时锁定相关行，其他事务必须等待
SELECT * FROM doctors WHERE status = 'on_duty' FOR UPDATE;
```
*   **效果**：T1 读取并锁定时，T2 必须等待。T1 提交后，T2 读取到最新状态（A 已下班），从而判断出自己不能下班。
*   **适用**：MySQL, TiDB, PostgreSQL。

### 2. 使用 SERIALIZABLE 隔离级别
将事务隔离级别提升到最高。
*   **PostgreSQL**：切换到 `SERIALIZABLE` 后，数据库会自动检测写偏斜并回滚冲突事务。
*   **SurrealDB / DynamoDB**：默认即为串行化，无需额外操作。

### 3. 应用层重试与乐观锁
*   在 PostgreSQL 的 SSI 模式下，捕获 `serialization failure` 错误并重试事务。
*   在 MongoDB 或 TiDB 中，可以在文档/行中增加版本号（Version），更新时检查版本号是否变更，若变更则重试。
