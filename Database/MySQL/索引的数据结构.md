## 概述

MySQL 的索引是其查询性能优化的核心机制。不同存储引擎采用不同的数据结构来实现索引，其中 **InnoDB**（MySQL 8.4 默认存储引擎）主要使用 **B-tree（实际为 B+Tree 变体）** 作为索引的基础数据结构。

## 索引数据结构类型

### B-Tree 索引（B+Tree 实现）

**适用引擎**：InnoDB、MyISAM

MySQL 官方文档中称为 "B-tree"，但 InnoDB 的实际实现是 **B+Tree** 变体。这是 MySQL 中最常用的索引数据结构，`PRIMARY KEY`、`UNIQUE`、普通 `INDEX` 和 `FULLTEXT`（InnoDB 使用倒排列表）都基于此结构。

**物理结构**：

- 索引记录存储在 B-tree 的**叶子节点页**（leaf pages）中
- 默认索引页大小为 **16KB**，由初始化时的 `innodb_page_size` 参数决定
- 新记录插入时，InnoDB 默认预留约 **1/16（约 6.25%）** 的页面空间用于后续插入和更新
- 顺序插入时，索引页约 **15/16 满**；随机插入时，页面填充率在 **1/2 到 15/16** 之间
- 当页面填充率低于 `MERGE_THRESHOLD`（默认 50%）时，InnoDB 会尝试收缩索引树以释放页面

**支持的查询操作**：

| 操作符 | 是否支持 | 说明 |
|--------|----------|------|
| `=` | ✅ | 精确匹配 |
| `>`、`>=`、`<`、`<=` | ✅ | 范围查询 |
| `BETWEEN` | ✅ | 范围查询 |
| `LIKE 'prefix%'` | ✅ | 前缀匹配（通配符不能在开头） |
| `LIKE '%pattern%'` | ❌ | 前导通配符无法使用索引（但字符串 > 3 字符时内部使用 Turbo Boyer-Moore 算法加速） |
| `ORDER BY` | ✅ | 当排序字段为索引左前缀时 |
| `GROUP BY` | ✅ | 当分组字段为索引左前缀时 |
| `IS NULL` | ✅ | 被索引列的 NULL 值查询可使用索引 |

**最左前缀原则**：

对于多列索引 `(col1, col2, col3)`，以下前缀可使用索引：
- `(col1)`
- `(col1, col2)`
- `(col1, col2, col3)`

但仅使用 `(col2)` 或 `(col2, col3)` 或 `(col3)` **无法使用索引**。

### Hash 索引

**适用引擎**：MEMORY（用户可选 B-tree 或 hash）、InnoDB（自适应哈希索引，Automatic Adaptive Hash Index）

**特性**：

- **仅支持等值比较**：`=` 和 `<=>` 操作符（速度极快）
- **不支持范围查询**：无法用于 `<`、`>`、`BETWEEN`、`ORDER BY` 等操作
- **不支持最左前缀**：必须使用完整键值查找
- **无法估算行数**：优化器无法判断两个值之间大约有多少行，影响执行计划选择

**InnoDB 自适应哈希索引（Adaptive Hash Index）**：

InnoDB 会在运行时监控索引查找模式，如果检测到某些页被频繁访问，会自动在内存中为这些页构建哈希索引。这是 InnoDB 的自动优化机制，无需人工干预。可通过 `innodb_adaptive_hash_index` 参数控制开关。

### R-Tree 索引

**适用场景**：空间数据类型（Spatial Data Types）

- 用于索引多维数据（如地理坐标、几何图形）
- InnoDB 和 MyISAM 的空间索引均使用 R-tree
- 通过 `SPATIAL INDEX` 创建
- **不支持 sorted index build**（排序索引构建）

### Full-Text 索引（倒排列表）

**适用引擎**：InnoDB（5.6+ 支持）

- InnoDB 的全文索引使用**倒排列表**（inverted list）而非 B-tree
- 用于全文搜索（`MATCH() ... AGAINST()` 语法）
- 支持自然语言搜索、布尔模式搜索和查询扩展搜索

## InnoDB 索引类型：聚簇索引与二级索引

### 聚簇索引（Clustered Index）

每个 InnoDB 表都有一个特殊的聚簇索引，**索引的叶子节点直接存储完整的行数据**。

**选择规则**（按优先级）：

1. 如果定义了 `PRIMARY KEY`，则使用它作为聚簇索引
2. 如果没有 `PRIMARY KEY`，则使用第一个所有列均为 `NOT NULL` 的 `UNIQUE` 索引
3. 如果以上都不满足，InnoDB 会自动生成一个隐藏的 `GEN_CLUST_INDEX`，基于一个 6 字节的自增 row ID

**性能优势**：

- 通过聚簇索引访问行时，索引搜索直接定位到包含行数据的页面
- 对于大表，相比"索引与数据分离存储"的方式，通常可节省一次磁盘 I/O

**最佳实践**：

- 每个表都应定义 `PRIMARY KEY`
- 如果没有逻辑上唯一且非空的列，建议添加自增列作为主键
- **主键应尽可能短**，因为二级索引的每个记录都会包含主键值，主键过长会浪费大量存储空间

### 二级索引（Secondary Index）

除聚簇索引外的所有索引都称为二级索引。

**关键机制**：

- 二级索引的每个记录包含两部分：**索引列的值 + 对应行的主键值**
- 通过二级索引查找行时，需要**两次索引查找**：先在二级索引中找到主键值，再用主键值到聚簇索引中查找完整行数据（称为"回表"）
- 因此主键越短，二级索引占用的空间越小

## B-Tree vs Hash 索引对比

| 特性 | B-Tree | Hash |
|------|--------|------|
| 等值查询 `=` | ✅ | ✅（极快） |
| 范围查询 `>`、`<`、`BETWEEN` | ✅ | ❌ |
| 前缀匹配 `LIKE 'abc%'` | ✅ | ❌ |
| `ORDER BY` 优化 | ✅ | ❌ |
| 最左前缀查找 | ✅ | ❌ |
| 行数估算（优化器使用） | ✅ | ❌ |
| 适用场景 | 通用查询 | 键值存储（key-value）风格的等值查找 |

## 索引填充因子与页面管理

### innodb_fill_factor

- 定义在 sorted index build（创建或重建 B-tree 索引）期间，每个 B-tree 页面被填充的百分比
- 剩余空间保留给未来的索引增长
- 设置为 100 时，聚簇索引页面仍会保留 1/16 的空间

### MERGE_THRESHOLD

- 默认值为 **50%**
- 当索引页面的填充率低于此阈值时，InnoDB 会尝试收缩索引树以释放页面
- 同时适用于 B-tree 和 R-tree 索引

## 覆盖索引（Covering Index）

当查询所需的所有列都包含在某个索引中时，MySQL 可以直接从索引树中获取结果，**无需回表查询数据行**。这种索引称为覆盖索引。

```sql
-- 假设在 (key_part1, key_part2, key_part3) 上建立了联合索引
SELECT key_part3 FROM tbl_name WHERE key_part1 = 1
-- 此查询可直接从索引树获取 key_part3，无需访问数据行
```

## 常见误区

### 误区 1：MySQL 的 B-tree 就是标准 B-tree

MySQL 文档中称为 "B-tree"，但 InnoDB 的实际实现是 **B+Tree** 变体。B+Tree 的特点是所有数据记录都存储在叶子节点，且叶子节点之间通过指针相连，这使得范围查询和顺序扫描更加高效。

### 误区 2：索引越多越好

- 不必要的索引浪费存储空间
- 每次 `INSERT`、`UPDATE`、`DELETE` 都需要更新所有相关索引，增加写入成本
- 优化器需要额外时间决定使用哪个索引
- 应在查询性能和写入开销之间找到平衡

### 误区 3：有索引就一定会使用

以下情况优化器可能选择全表扫描而非索引：

- 估计需要访问表中很大比例的行（此时全表扫描的磁盘寻道次数更少）
- 但如果查询使用了 `LIMIT`，即使访问比例很大，优化器仍可能使用索引（因为能更快找到需要的少量行）

### 误区 4：LIKE 查询完全不能用索引

- `LIKE 'prefix%'`（前缀匹配）**可以使用索引**
- `LIKE '%pattern%'`（包含匹配）**不能使用索引**，但当 pattern 长度 > 3 时，MySQL 内部使用 Turbo Boyer-Moore 算法加速搜索

## 索引使用场景总结

MySQL 在以下场景中会使用索引：

1. 快速查找匹配 `WHERE` 条件的行
2. 多索引选择时，选择能过滤出最少行数的索引（最具选择性的索引）
3. 多列索引的左前缀查找
4. JOIN 操作中关联其他表的列（关联列类型和大小一致时效率更高）
5. 查找索引列的 `MIN()` 或 `MAX()` 值
6. 对索引左前缀进行 `ORDER BY` 或 `GROUP BY` 操作
7. 覆盖索引场景下，直接从索引树获取查询结果

## 注意事项

- **字符集一致性**：JOIN 或比较操作中，非二进制字符串列应使用相同字符集（如 `utf8mb4` 与 `latin1` 比较会导致索引失效）
- **类型一致性**：字符串列与数值列比较时，由于隐式转换可能导致多种匹配结果，索引无法使用
- **小表场景**：对于小表或需要访问大部分行的查询，全表扫描可能比索引查找更快（减少磁盘寻道）
- **VARCHAR 与 CHAR**：在 JOIN 场景中，相同声明大小的 `VARCHAR(10)` 和 `CHAR(10)` 被视为相同类型，但 `VARCHAR(10)` 和 `CHAR(15)` 不是
