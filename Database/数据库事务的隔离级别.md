# 数据库事务的隔离级别

---

## 是什么

事务隔离级别（Transaction Isolation Level）是数据库 ACID 属性中 **I（Isolation）** 的具体实现粒度，用于控制多个并发事务之间的可见性边界。隔离级别越高，并发事务相互干扰的可能性越低，但系统资源开销越大、并发性能越差；隔离级别越低，并发性能越好，但可能引发脏读、不可重复读、幻读等并发异常。

SQL 标准（ANSI/ISO SQL-92）定义了四种标准隔离级别，从低到高依次为：

1. `READ UNCOMMITTED`（读未提交）
2. `READ COMMITTED`（读已提交）
3. `REPEATABLE READ`（可重复读）
4. `SERIALIZABLE`（可串行化）

---

## 为什么重要

在并发场景下，多个事务同时读写相同数据时会产生三类典型的**读异常（Read Phenomena）**：

| 异常类型 | 英文 | 描述 |
|---------|------|------|
| **脏读** | Dirty Read | 一个事务读取了另一个事务**尚未提交**的修改。若后者回滚，前者读到的数据即为无效数据。 |
| **不可重复读** | Non-repeatable Read | 一个事务在同一事务内两次读取同一行数据，结果不一致（因为中间有其他事务**修改并提交**了该行）。 |
| **幻读** | Phantom Read | 一个事务在同一事务内两次执行相同范围查询，结果集的行数不一致（因为中间有其他事务**插入或删除**了符合查询条件的行）。 |

隔离级别的核心作用就是**定义哪些异常被允许发生、哪些被禁止**，从而在一致性与性能之间做出权衡。

---

## 四种隔离级别详解

### 1. READ UNCOMMITTED（读未提交）

- **最低隔离级别**
- 允许脏读、不可重复读、幻读全部发生
- 事务可以读取其他事务尚未提交的修改
- 实际应用中极少使用，仅适用于对数据一致性要求极低的场景（如实时统计大盘的近似值）

```sql
-- MySQL 设置方式
SET SESSION TRANSACTION ISOLATION LEVEL READ UNCOMMITTED;
```

### 2. READ COMMITTED（读已提交）

- 禁止脏读，允许不可重复读和幻读
- 保证每次读取的数据都是其他事务已提交的版本
- 同一事务内多次读取同一行，可能得到不同结果（因为中间有其他事务提交了对该行的修改）
- **PostgreSQL、Oracle 的默认隔离级别**

```sql
-- MySQL 设置方式
SET SESSION TRANSACTION ISOLATION LEVEL READ COMMITTED;
```

**InnoDB 实现特点**（MySQL）：
- 每次一致读（consistent read）都会创建并读取自己的新快照
- 只锁住被修改或删除的行，不锁间隙（gap），因此新记录可以插入到已锁定行之间
- 仅支持行格式的二进制日志（row-based binary logging）

### 3. REPEATABLE READ（可重复读）

- 禁止脏读和不可重复读，**允许幻读**（SQL 标准定义）
- 同一事务内的多次读取返回一致的快照
- **MySQL InnoDB 的默认隔离级别**

```sql
-- MySQL 设置方式（默认值，通常无需显式设置）
SET SESSION TRANSACTION ISOLATION LEVEL REPEATABLE READ;
```

**InnoDB 实现特点**（MySQL）：
- 一致读基于事务内第一次读取时建立的快照，后续读取都基于该快照
- 使用 **next-key 锁**（记录锁 + 间隙锁）锁定扫描范围，**实际上在大多数场景下也能避免幻读**（这是 InnoDB 对 SQL 标准的增强）
- 对于唯一索引的精确查找，只锁记录不锁间隙
- 对于范围查找，使用 gap 锁或 next-key 锁阻止其他会话在范围内插入新记录

**注意事项**：
- 不建议在同一 `REPEATABLE READ` 事务中混用锁读（`SELECT ... FOR UPDATE`）和无锁读（普通 `SELECT`），因为两者看到的数据状态可能不一致

### 4. SERIALIZABLE（可串行化）

- **最高隔离级别**
- 禁止脏读、不可重复读和幻读
- 强制事务串行执行，等价于事务一个接一个地顺序执行
- 并发性能最低，通常仅用于对一致性要求极高的场景（如金融账务、XA 事务）

```sql
-- MySQL 设置方式
SET SESSION TRANSACTION ISOLATION LEVEL SERIALIZABLE;
```

**InnoDB 实现特点**（MySQL）：
- 在 `autocommit` 关闭时，隐式将所有普通 `SELECT` 转换为 `SELECT ... FOR SHARE`（共享锁读）
- 在 `autocommit` 开启时，`SELECT` 是独立事务，可串行化执行

## 隔离级别与并发异常对照表

| 隔离级别 | 脏读 | 不可重复读 | 幻读 |
|---------|------|-----------|------|
| `READ UNCOMMITTED` | ✅ 允许 | ✅ 允许 | ✅ 允许 |
| `READ COMMITTED` | ❌ 禁止 | ✅ 允许 | ✅ 允许 |
| `REPEATABLE READ` | ❌ 禁止 | ❌ 禁止 | ✅ 允许（SQL 标准） |
| `SERIALIZABLE` | ❌ 禁止 | ❌ 禁止 | ❌ 禁止 |

> **说明**：MySQL InnoDB 在 `REPEATABLE READ` 下通过 next-key 锁机制，实际上也能在大多数场景下避免幻读，但这属于实现层面的增强，不等同于 SQL 标准定义。

## 各数据库实现的隔离级别

### MySQL (InnoDB)

- **支持的隔离级别**：全部四种标准隔离级别
- **默认隔离级别**：`REPEATABLE READ`
- **实现机制**：MVCC（多版本并发控制） + Next-Key 锁
- **特点**：
  - 通过 MVCC 实现一致的无锁读（consistent nonlocking reads）
  - 通过 Next-Key 锁（记录锁 + 间隙锁）在 `REPEATABLE READ` 下实际避免了大多数幻读场景
  - 可通过 `SET TRANSACTION` 语句在会话或事务级别动态调整

```sql
-- 查看当前隔离级别
SELECT @@transaction_isolation;

-- 设置会话隔离级别
SET SESSION TRANSACTION ISOLATION LEVEL READ COMMITTED;
```

### PostgreSQL

- **支持的隔离级别**：`READ COMMITTED`、`REPEATABLE READ`、`SERIALIZABLE`
- **默认隔离级别**：`READ COMMITTED`
- **不支持**：`READ UNCOMMITTED`（PostgreSQL 中该级别实际行为等同于 `READ COMMITTED`）
- **实现机制**：MVCC（多版本并发控制） + SSI（Serializable Snapshot Isolation）
- **特点**：
  - PostgreSQL 的 `REPEATABLE READ` 基于快照隔离（Snapshot Isolation），完全避免幻读（比 SQL 标准更强）
  - `SERIALIZABLE` 级别使用 SSI 技术，通过检测读写冲突来保证串行化，而非传统的锁机制
  - 若检测到序列化冲突，会抛出 `serialization failure` 错误，需要应用层重试

```sql
-- 查看当前隔离级别
SHOW transaction_isolation;

-- 设置事务隔离级别
SET TRANSACTION ISOLATION LEVEL SERIALIZABLE;
```

### SurrealDB

- **支持的隔离级别**：`SERIALIZABLE`（默认且唯一）
- **默认隔离级别**：`SERIALIZABLE`
- **实现机制**：MVCC + 多版本快照存储（Multi-version snapshot storage）
- **特点**：
  - SurrealDB 采用**多版本键值存储**，为读取提供快照隔离（snapshot isolation），事务可以在一致的时间点视图上运行，无需阻塞并发写入或全局协调[^1]
  - 官方明确声明：**"Native ACID compliant"**，保证数据的即时一致性（immediate data consistency）[^2]
  - 每个语句默认在独立事务中运行，也可通过 `BEGIN TRANSACTION` / `COMMIT TRANSACTION` 显式组合多个语句为单一事务[^3]
  - 支持跨多行、多表的长事务，无事务时长限制
  - 分布式模式下采用**快速路径共识协议**（fast-path consensus），大多数事务在单次网络往返内提交，仅冲突事务回退到多轮慢路径[^4]
  - 企业版/云版支持**乐观并发控制**（optimistic concurrency control），通过 quorum 协议提交事务，瞬态冲突自动重试[^5]

```sql
-- SurrealDB 事务示例
BEGIN TRANSACTION;

UPDATE account:one SET balance += 300.00;
UPDATE account:two SET balance -= 300.00;

COMMIT TRANSACTION;
```

> **注意**：SurrealDB 目前不提供隔离级别的选择（如 `READ COMMITTED` 或 `REPEATABLE READ`），所有事务默认以 `SERIALIZABLE` 级别执行。这是其设计哲学——优先保证一致性，而非提供一致性/性能的权衡选项。

### TiDB

- **支持的隔离级别**：`READ COMMITTED`、`REPEATABLE READ`
- **默认隔离级别**：`REPEATABLE READ`
- **实现机制**：MVCC + 乐观/悲观事务模型
- **特点**：
  - **REPEATABLE READ**：TiDB 实现了**快照隔离（Snapshot Isolation, SI）**，对外宣称兼容 MySQL 的 `REPEATABLE READ`。与 ANSI 标准不同，SI 禁止严格幻读（Strict Phantoms），但允许写偏斜（Write Skew）[^6]。
  - **READ COMMITTED**：自 v4.0 起支持，但**仅在悲观事务模式下生效**。在乐观模式下设置无效，仍回退到 Repeatable Read[^7]。
  - **事务模型**：支持乐观（Optimistic）和悲观（Pessimistic）两种事务模型。悲观模式下默认使用 `REPEATABLE READ`，且当前读（`SELECT ... FOR UPDATE`）不可重复。

```sql
-- 查看当前隔离级别
SHOW VARIABLES LIKE 'transaction_isolation';

-- 设置会话隔离级别
SET SESSION transaction_isolation = 'READ-COMMITTED';
```

### MongoDB

- **支持的隔离级别**：`Snapshot Isolation`（多文档事务）
- **默认隔离级别**：`Snapshot Isolation`
- **实现机制**：MVCC + 快照隔离
- **特点**：
  - **单文档原子性**：MongoDB 单个文档的操作天然原子，无需事务。
  - **多文档事务**：自 4.0（副本集）和 4.2（分片集群）起支持。事务内的所有操作基于**事务开始时的快照**，提供快照隔离级别[^8]。
  - **异常表现**：快照隔离禁止脏读和不可重复读，但理论上允许写偏斜。
  - **重试机制**：驱动程序通常内置 `TransientTransactionError` 的自动重试逻辑。

```javascript
// MongoDB Node.js Driver 示例
session.withTransaction(async () => {
    await collection1.insertOne(session, doc1);
    await collection2.updateOne(session, filter, update);
});
```

### DynamoDB

- **支持的隔离级别**：`SERIALIZABLE`（事务 API）
- **默认隔离级别**：`SERIALIZABLE`（针对 `TransactWriteItems` / `TransactGetItems`）
- **实现机制**：两阶段提交（2PC）变体 + 锁
- **特点**：
  - DynamoDB 的事务 API（Transactions）提供完整的 ACID 保证，隔离级别等效于**串行化**[^9]。
  - 事务内的读写操作对其他并发事务不可见，直到事务提交。
  - 底层通过准备（Prepare）和提交（Commit）两个阶段保证一致性，期间会对涉及项加锁。

```javascript
// DynamoDB TransactWriteItems 示例
await dynamodb.transactWriteItems({
    TransactItems: [
        { Update: { TableName: 'Table1', Key: {...}, UpdateExpression: '...' } },
        { Put: { TableName: 'Table2', Item: {...} } }
    ]
}).promise();
```

### 对比总结

| 数据库 | 支持的隔离级别 | 默认级别 | 实现机制 |
|--------|---------------|---------|---------|
| **MySQL (InnoDB)** | `READ UNCOMMITTED`, `READ COMMITTED`, `REPEATABLE READ`, `SERIALIZABLE` | `REPEATABLE READ` | MVCC + Next-Key 锁 |
| **PostgreSQL** | `READ COMMITTED`, `REPEATABLE READ`, `SERIALIZABLE` | `READ COMMITTED` | MVCC + SSI |
| **TiDB** | `READ COMMITTED`, `REPEATABLE READ` | `REPEATABLE READ` | MVCC + SI (悲观/乐观) |
| **MongoDB** | `Snapshot Isolation` (多文档事务) | `Snapshot Isolation` | MVCC + 快照 |
| **DynamoDB** | `SERIALIZABLE` (事务 API) | `SERIALIZABLE` | 2PC 变体 + 锁 |
| **SurrealDB** | `SERIALIZABLE`（唯一） | `SERIALIZABLE` | MVCC + 多版本快照存储 |
| Oracle | `READ COMMITTED`, `SERIALIZABLE` | `READ COMMITTED` | MVCC |
| SQL Server | `READ UNCOMMITTED`, `READ COMMITTED`, `REPEATABLE READ`, `SERIALIZABLE`, `SNAPSHOT` | `READ COMMITTED` | 锁 + 行版本控制 |
| SQLite | `SERIALIZABLE`（单连接） | `SERIALIZABLE` | 文件级锁 |

## 并发控制机制

数据库实现隔离级别主要依赖两种并发控制策略：

### 1. 基于锁的并发控制（Lock-based Concurrency Control, LBCC）

- 事务访问数据前必须先获取锁
- 锁类型：共享锁（S 锁，读锁）、排他锁（X 锁，写锁）、间隙锁（Gap Lock）、Next-Key 锁
- **两阶段锁协议（2PL）**：加锁阶段只能加锁不能解锁，解锁阶段只能解锁不能加锁
- 优点：实现成熟、一致性强
- 缺点：锁竞争导致性能下降，可能引发死锁

### 2. 多版本并发控制（Multiversion Concurrency Control, MVCC）

- 数据库维护数据的多个历史版本
- 读操作读取快照（历史版本），写操作创建新版本
- 读写不冲突，大幅提升并发性能
- **PostgreSQL、Oracle、MySQL InnoDB、TiDB、MongoDB、SurrealDB 均采用 MVCC**
- 缺点：存储开销增加，需要定期清理旧版本（vacuum / purge）

## 常见误区

### 误区 1：REPEATABLE READ 完全不会发生幻读

- **SQL 标准定义**：`REPEATABLE READ` 允许幻读
- **MySQL InnoDB 实现**：通过 next-key 锁在大多数场景下避免了幻读，但并非绝对
- **PostgreSQL 实现**：基于快照隔离，`REPEATABLE READ` 实际上也避免了幻读（比 SQL 标准更强）
- **TiDB 实现**：实现的是快照隔离（SI），禁止严格幻读，但允许写偏斜
- 若需要严格避免幻读和写偏斜，应使用 `SERIALIZABLE`

### 误区 2：隔离级别越高越好

- 高隔离级别意味着更多锁竞争、更低并发、更高死锁概率
- 应根据业务场景选择：
  - 报表统计、近似计算 → `READ COMMITTED` 甚至 `READ UNCOMMITTED`
  - 一般业务 CRUD → `READ COMMITTED` 或 `REPEATABLE READ`
  - 金融账务、强一致性场景 → `SERIALIZABLE`
- **SurrealDB / DynamoDB 的设计选择**：默认 `SERIALIZABLE`，适用于对一致性要求高的场景，但在高并发写入场景可能需要关注冲突重试或吞吐量限制

### 误区 3：ANSI SQL 隔离级别定义完全准确

- ANSI SQL-92 基于"异常现象"定义隔离级别，但存在歧义
- 论文 *A Critique of ANSI SQL Isolation Levels* 指出：三种 ANSI 异常现象既不充分也不必要，无法准确刻画实际数据库的隔离行为
- 现代数据库（如 PostgreSQL 的 SSI、TiDB 的 SI、MongoDB 的快照隔离）的实际隔离行为与 ANSI 定义存在差异

### 误区 4：所有数据库都支持四种标准隔离级别

- **PostgreSQL 不支持 `READ UNCOMMITTED`**：该级别在 PostgreSQL 中实际行为等同于 `READ COMMITTED`
- **TiDB 不支持 `SERIALIZABLE`**：TiDB 最高支持到 `REPEATABLE READ`（实际上是 SI），不提供标准的 `SERIALIZABLE` 级别
- **MongoDB / DynamoDB**：作为 NoSQL 数据库，不直接使用 SQL 标准术语，而是通过事务 API 提供特定的隔离保证（如快照隔离或串行化）

## 设置语法示例

### MySQL

```sql
-- 查看当前隔离级别
SELECT @@transaction_isolation;

-- 设置全局隔离级别（影响新连接）
SET GLOBAL TRANSACTION ISOLATION LEVEL READ COMMITTED;

-- 设置当前会话隔离级别
SET SESSION TRANSACTION ISOLATION LEVEL READ COMMITTED;

-- 在事务内设置（仅影响当前事务）
START TRANSACTION;
SET TRANSACTION ISOLATION LEVEL SERIALIZABLE;
-- ... 事务操作 ...
COMMIT;
```

### PostgreSQL

```sql
-- 查看当前隔离级别
SHOW transaction_isolation;

-- 设置当前事务隔离级别
SET TRANSACTION ISOLATION LEVEL SERIALIZABLE;

-- 或在事务开始时指定
BEGIN ISOLATION LEVEL REPEATABLE READ;
```

### TiDB

```sql
-- 查看当前隔离级别
SHOW VARIABLES LIKE 'transaction_isolation';

-- 设置当前会话隔离级别（仅在悲观模式下生效 RC）
SET SESSION transaction_isolation = 'READ-COMMITTED';
```

### MongoDB

```javascript
// MongoDB 不提供 SQL 级别的设置，事务默认使用快照隔离
// 通过驱动程序的 withTransaction 或 startTransaction 使用
const session = client.startSession();
session.withTransaction(async () => {
    // 操作自动在快照隔离下执行
    await db.collection.insertOne(doc, { session });
});
```

### SurrealDB

```sql
-- SurrealDB 不提供隔离级别设置，所有事务默认 SERIALIZABLE
-- 显式事务示例
BEGIN TRANSACTION;
  UPDATE account:one SET balance += 100;
  UPDATE account:two SET balance -= 100;
COMMIT TRANSACTION;
```

## 相关概念

- **ACID**：原子性（Atomicity）、一致性（Consistency）、隔离性（Isolation）、持久性（Durability）
- **快照隔离（Snapshot Isolation）**：基于 MVCC 的隔离实现，提供比 `READ COMMITTED` 更强、比 `SERIALIZABLE` 更弱的隔离保证
- **写偏斜（Write Skew）**：在 `REPEATABLE READ` 或 `Snapshot Isolation` 下可能发生的一种异常，两个事务各自读取并修改不同但相关的行，导致提交后数据不一致（PostgreSQL 的 SSI 可检测并阻止）
- **丢失更新（Lost Update）**：两个事务同时修改同一行，后提交的覆盖先提交的结果（`READ COMMITTED` 及以上级别通过写锁避免）
