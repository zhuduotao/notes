## 是什么

浏览器资源加载顺序是指浏览器在解析 HTML 文档时，如何决定不同类型资源（CSS、JavaScript、图片、字体等）的**下载优先级**和**执行顺序**。这涉及两个层面：

1. **浏览器内部优先级系统**：浏览器为每种资源类型分配默认优先级，决定网络队列中的下载顺序
2. **开发者控制机制**：通过 HTML 属性（`async`、`defer`、`fetchpriority`）和资源提示（`preload`、`preconnect` 等）干预默认行为

理解资源加载顺序是前端性能优化的核心基础，直接影响 LCP（最大内容绘制）、FCP（首次内容绘制）等 Core Web Vitals 指标。

---

## 为什么重要

- **LCP 优化关键**：LCP 元素（通常是首屏图片或大块文本）的资源优先级直接影响 LCP 分数。浏览器可能无法自动识别哪个资源对 LCP 最重要
- **避免资源竞争**：不当的加载顺序会导致关键资源被非关键资源阻塞，延长页面可交互时间
- **带宽利用率**：HTTP/1.1 下并发连接数有限（通常 6 个），优先级设置决定哪些资源先获得连接
- **用户体验**：正确的加载顺序让用户更快看到有意义的内容，减少感知加载时间

---

## 浏览器内部资源优先级系统

浏览器为每种资源类型分配内部优先级，决定网络请求队列中的下载顺序。以下是 Chromium 浏览器的默认优先级（不同浏览器实现可能不同）：

### 默认优先级表

| 资源类型 | 默认优先级 | 说明 |
|----------|-----------|------|
| `<script>`（渲染阻塞） | Highest | 位于 `<head>` 中无 `async`/`defer` 的脚本 |
| `<link rel="stylesheet">` | Highest | 样式表阻塞渲染，优先级最高 |
| `<img>`（首屏内） | High | 视口内的图片 |
| `<img>`（首屏外） | Low | 视口外的图片（延迟加载） |
| `<script defer>` | Low | 延迟执行的脚本 |
| `<script async>` | Low | 异步脚本 |
| 字体（`@font-face`） | High | 首次引用的字体 |
| 字体（后续引用） | Low | 已缓存字体的变体 |
| `<link rel="preload">` | Highest | 显式预加载的资源 |
| `<link rel="prefetch">` | Lowest | 未来页面可能需要的资源 |
| `<link rel="dns-prefetch">` | Lowest | DNS 预解析 |
| `<link rel="preconnect">` | Highest | 提前建立连接 |

> **注意**：以上是 Chromium 的实现细节，不是 Web 标准。Firefox 和 Safari 的优先级系统类似但具体值可能不同。来源：[Chrome DevTools Network 面板优先级列](https://developer.chrome.com/docs/devtools/network/reference/#priority)

### 优先级动态调整

浏览器会根据页面解析进度动态调整资源优先级：

- 图片初始可能是 Low，当滚动到视口附近时提升为 High
- 字体首次加载为 High，后续变体为 Low
- `<link rel="preload">` 的资源始终为 Highest

## 开发者控制机制

### `fetchpriority` 属性（Priority Hints）

`fetchpriority` 是 HTML 新增的全局属性（Baseline 2024，2024 年 10 月起广泛支持），允许开发者显式提示浏览器调整资源获取优先级[^mdn-fetchpriority]。

**支持元素**：`<img>`、`<link>`、`<script>`

**可选值**：

| 值 | 效果 |
|----|------|
| `high` | 相对于其他资源提高获取优先级 |
| `low` | 相对于其他资源降低获取优先级 |
| `auto` | 不设置偏好（默认值） |

**典型用例**：

```html
<!-- 提升 LCP 图片优先级 -->
<img src="hero.jpg" fetchpriority="high" alt="Hero image">

<!-- 降低首屏外图片优先级 -->
<img src="below-fold.jpg" fetchpriority="low" alt="Secondary content">

<!-- 与 preload 配合使用，进一步提升关键资源 -->
<link rel="preload" href="critical-font.woff2" as="font" fetchpriority="high" crossorigin>
```

**注意事项**：

- 该属性是**提示（hint）**而非指令，浏览器可能忽略
- 应**谨慎使用**，过度或错误的优先级设置可能降低性能
- 对 `async`/`defer` 脚本设置 `fetchpriority` 影响有限，因为它们的执行时机已受属性控制

[^mdn-fetchpriority]: 来源：[MDN - fetchpriority](https://developer.mozilla.org/en-US/docs/Web/HTML/Reference/Attributes/fetchpriority)

### `<script>` 加载控制

`async` 和 `defer` 是控制脚本加载和执行时机的核心属性。详见 [[HTML script 标签 async 与 defer 属性详解]]。

**快速对比**：

| 属性 | 下载 | 执行时机 | 执行顺序 |
|------|------|---------|---------|
| 默认 | 阻塞解析 | 下载完立即执行 | 按出现顺序 |
| `async` | 并行 | 下载完成后（可能阻塞解析） | 不确定 |
| `defer` | 并行 | DOM 解析完成后 | 按出现顺序 |
| `type="module"` | 并行 | DOM 解析完成后（自带 defer） | 按出现顺序 |

### 资源提示（Resource Hints）

通过 `<link>` 元素提示浏览器提前准备资源或连接：

| 提示类型 | 用途 | 优先级 | 适用场景 |
|----------|------|--------|---------|
| `<link rel="preconnect">` | 提前建立 DNS/TCP/TLS 连接 | Highest | 第三方资源域名（CDN、字体服务） |
| `<link rel="dns-prefetch">` | 仅提前解析 DNS | Lowest | 仅需 DNS 解析的场景 |
| `<link rel="preload">` | 预加载当前页面必需资源 | Highest | LCP 图片、关键字体、关键 CSS/JS |
| `<link rel="prefetch">` | 预加载未来可能需要的资源 | Lowest | 下一页资源、用户可能点击的链接目标 |
| `<link rel="modulepreload">` | 预加载 ES Module 及其依赖 | Highest | 模块化为 ES Module 的应用 |
| `<link rel="prerender">` | 预渲染整个页面 | Highest（谨慎） | 高置信度的下一页导航 |

**`preload` 使用示例**：

```html
<!-- 预加载关键字体（需 crossorigin） -->
<link rel="preload" href="/fonts/critical.woff2" as="font" type="font/woff2" crossorigin>

<!-- 预加载 LCP 图片 -->
<link rel="preload" href="/images/hero.webp" as="image">

<!-- 预加载关键 CSS -->
<link rel="preload" href="/css/critical.css" as="style">
```

> **重要**：`preload` 的 `as` 属性必须正确设置，否则浏览器会使用错误的优先级和加载策略。来源：[MDN - preload](https://developer.mozilla.org/en-US/docs/Web/HTML/Reference/Attributes/rel/preload)

## 资源加载顺序完整流程

以下是浏览器从接收 HTML 到资源加载完成的典型流程：

```
1. HTML 文档开始下载
   ↓
2. 预加载扫描器（Preload Scanner）启动
   - 与 HTML 解析器并行运行
   - 提前发现 <img>、<link>、<script> 等资源并发起请求
   ↓
3. HTML 解析器逐行解析
   - 遇到 <link rel="stylesheet"> → 下载并阻塞渲染
   - 遇到 <script>（无属性） → 暂停解析，下载并执行
   - 遇到 <script defer> → 后台下载，延迟到 DOM 完成后执行
   - 遇到 <script async> → 后台下载，完成后立即执行
   - 遇到 <img> → 根据是否在视口内分配优先级
   ↓
4. CSSOM 构建完成后，与 DOM 合并生成 Render Tree
   ↓
5. 布局（Layout）→ 绘制（Paint）→ 合成（Composite）
   ↓
6. defer 脚本按顺序执行
   ↓
7. DOMContentLoaded 事件触发
   ↓
8. 其他资源（图片、prefetch 等）继续加载
   ↓
9. window.onload 事件触发
```

> 预加载扫描器是性能优化的关键机制，它让资源下载不需要等待主线程解析到对应标签。详见 [[渲染管线]] 中"预加载扫描器"部分。

## 不同类型资源的加载策略

### CSS

- **默认行为**：`<link rel="stylesheet">` 是渲染阻塞资源（render-blocking），浏览器会等待 CSSOM 构建完成才渲染
- **优化策略**：
  - 内联关键 CSS（Critical CSS）到 `<head>`
  - 非关键 CSS 使用 `media` 属性延迟加载：
    ```html
    <link rel="stylesheet" href="non-critical.css" media="print" onload="this.media='all'">
    ```
  - 使用 `preload` 预加载关键 CSS

### JavaScript

- **默认行为**：`<script>` 阻塞 HTML 解析
- **优化策略**：
  - 优先使用 `defer`（适用于大多数应用脚本）
  - 独立第三方脚本使用 `async`
  - 使用 `type="module"` 自带 defer 行为
  - 代码分割 + 动态 `import()` 按需加载

### 图片

- **默认行为**：浏览器根据图片是否在视口内分配优先级
- **优化策略**：
  - LCP 图片使用 `fetchpriority="high"` 或 `<link rel="preload">`
  - 首屏外图片使用 `loading="lazy"`
  - 使用现代格式（WebP、AVIF）和响应式图片（`srcset`）
  - 为图片设置 `width` 和 `height` 避免 CLS

### 字体

- **默认行为**：首次引用的字体为 High 优先级，后续变体为 Low
- **优化策略**：
  - 使用 `preload` 预加载关键字体
  - 使用 `font-display: swap` 或 `optional` 控制显示行为
  - 字体子集化减少文件体积
  - 使用 `crossorigin` 属性（字体预加载必须）

## HTTP/2 和 HTTP/3 对加载顺序的影响

### HTTP/1.1 的限制

- 每个域名最多 **6 个并发连接**
- 资源必须排队等待可用连接
- 优先级设置尤为重要，决定哪些资源先获得连接

### HTTP/2 的改进

- 单一连接上的**多路复用**，无并发连接数限制
- 引入**服务器推送（Server Push）**（但已被 Chrome 废弃，2022 年移除）
- 支持**优先级树（Priority Tree）**，客户端可告知服务器资源依赖关系

### HTTP/3 的进一步改进

- 基于 QUIC 协议，解决 TCP 队头阻塞问题
- 优先级机制与 HTTP/2 类似，但传输层更可靠
- 弱网环境下表现更好

> **注意**：即使使用 HTTP/2/3，浏览器内部仍有优先级系统。多路复用解决了连接数限制，但不改变资源优先级决定下载顺序的事实。

## 常见误区

### 误区 1：`preload` 越多越好

**错误**。过度使用 `preload` 会：
- 浪费带宽预加载不需要的资源
- 挤占其他关键资源的带宽
- 增加内存占用

**正确做法**：仅预加载对当前页面渲染**至关重要**的资源（LCP 元素、关键字体、关键 CSS）。

### 误区 2：`fetchpriority="high"` 会让资源最先加载

**不完全正确**。`fetchpriority` 是**相对优先级提示**，不是绝对指令。浏览器可能忽略该提示，特别是当：
- 资源类型本身优先级已经很高
- 网络条件不允许
- 浏览器实现策略不同

### 误区 3：`async` 脚本完全不阻塞

**错误**。`async` 脚本**下载时不阻塞**解析，但**执行时会短暂阻塞**解析。只是阻塞时间通常很短。

### 误区 4：`defer` 脚本在 `window.onload` 之后执行

**错误**。`defer` 脚本在 HTML 解析完成后、`DOMContentLoaded` 事件**之前**执行。`window.onload` 要等到所有资源加载完毕才触发。

### 误区 5：图片 `loading="lazy"` 对所有图片都适用

**错误**。LCP 图片**不应**使用 `loading="lazy"`，因为这会延迟 LCP 图片加载，直接损害 LCP 分数。仅对首屏外图片使用懒加载。

## 最佳实践

### 首屏加载优化清单

1. **识别关键资源**：使用 Chrome DevTools Network 面板查看资源优先级和加载瀑布图
2. **内联关键 CSS**：将首屏必需样式内联到 `<head>`
3. **延迟非关键 JS**：使用 `defer` 或动态 `import()`
4. **预加载 LCP 资源**：使用 `<link rel="preload">` 或 `fetchpriority="high"`
5. **预连接第三方域名**：对 CDN、字体服务使用 `<link rel="preconnect">`
6. **优化图片**：现代格式 + 响应式 + 正确的懒加载策略
7. **字体优化**：预加载关键字体 + `font-display` 策略

### 调试工具

| 工具 | 用途 |
|------|------|
| Chrome DevTools Network 面板 | 查看资源优先级（Priority 列）、加载瀑布图 |
| Chrome DevTools Performance 面板 | 分析资源加载对渲染的影响 |
| Lighthouse | 自动审计资源加载问题 |
| WebPageTest | 多地点、多网络的详细加载分析 |
