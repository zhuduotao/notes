## 是什么

浏览器网络请求优先级（Resource Prioritization）是浏览器内核在加载网页时，为每个子资源（CSS、JavaScript、图片、字体等）自动分配的下载优先级机制。浏览器通过优先级决定**哪些资源先下载、哪些后下载、哪些可以并行**，从而在有限带宽下优化页面加载体验。

开发者可以通过 Fetch Priority API（`fetchpriority` 属性）和 Resource Hints（`preload`、`preconnect` 等）干预浏览器的默认优先级决策。

## 为什么重要

- **带宽竞争**：在 HTTP/1.x 下并行连接数有限（通常 6 个），HTTP/2/3 虽然支持多路复用但共享同一 TCP/QUIC 连接的带宽。优先级决定了有限带宽的分配顺序。
- **Core Web Vitals 影响**：LCP（最大内容绘制）直接受 LCP 资源（通常是图片）的加载优先级影响。正确设置优先级可显著降低 LCP 时间。
- **浏览器默认策略不总是最优**：浏览器基于启发式规则分配优先级，但无法理解业务语义（例如轮播图中只有第一张图是首屏可见的，但浏览器不知道）。
- **Preload Scanner 协同**：优先级与 Preload Scanner（预加载扫描器）配合工作，理解两者关系才能避免"对抗浏览器优化"的反模式。

## 浏览器默认优先级机制

### 五级优先级体系

Chrome（Blink 内核）使用五级优先级调度资源下载：

| Blink 内部优先级 | DevTools 显示名称 | 说明 |
|---|---|---|
| `VeryHigh` | **Highest** | 最高优先级，布局阻塞阶段立即下载 |
| `High` | **High** | 高优先级，布局阻塞阶段下载 |
| `Medium` | **Medium** | 中等优先级，布局阻塞阶段逐个下载 |
| `Low` | **Low** | 低优先级，布局阻塞阶段后下载 |
| `VeryLow` | **Lowest** | 最低优先级，最后下载 |

> **注意**：DevTools Network 面板中显示的优先级名称与 Blink 内部名称略有不同（`VeryHigh` → `Highest`，`VeryLow` → `Lowest`）。

### 各资源类型的默认优先级

以下表格展示 Chrome 如何为不同资源类型分配优先级（基于资源在文档中的位置和属性）：

| 资源类型 | 默认优先级 | 是否在布局阻塞阶段下载 | 说明 |
|---|---|---|---|
| 主文档（HTML） | Highest | — | 导航请求 |
| CSS（早期，`<head>` 中） | Highest | 是 | 阻塞渲染，优先级最高 |
| CSS（晚期，`<body>` 中） | High | 是（逐个） | 不阻塞初始布局 |
| CSS（media 不匹配，如 `media="print"`） | Lowest | 否 | 不会被 Preload Scanner 发现 |
| Script（早期或非 Preload Scanner 发现） | Highest | 是 | 阻塞 HTML 解析 |
| Script（晚期） | High | 是（逐个） | 阻塞 HTML 解析 |
| Script（`async`/`defer`） | Low | 否 | 不阻塞解析，但默认低优先级 |
| Font | Highest | 是 | 字体默认最高优先级 |
| Font（`rel="preload"`） | Highest | 是 | 预加载字体 |
| Image（视口内，布局后提升） | High | 初始 Low，布局后提升 | 布局完成后检测是否在视口内并提升 |
| Image（前 5 张大图，>10000px²） | Medium | 是（Chrome 117+） | 自动提升前 5 张大图优先级 |
| Image（普通） | Low | 否 | 默认低优先级 |
| Media（video/audio） | Low | 否 | 媒体资源默认低优先级 |
| XHR/Fetch（异步） | High | 是 | `fetch()` 默认高优先级 |
| Prefetch | Lowest | 否 | 预取未来页面资源 |

> **"早期"与"晚期"的定义**：在任何非预加载的图片被请求之前请求的资源为"早期"（early），之后请求的为"晚期"（late）。

### 优先级动态变化

浏览器会在页面加载过程中动态调整资源优先级：

- **图片优先级提升**：图片初始为 `Low`，布局完成后浏览器检测其是否在视口内，如果在则提升为 `High`。这个提升过程会产生延迟。
- **Chrome 117 优化**：前 5 张面积大于 10000px² 的图片自动设为 `Medium` 优先级，其中 2 张可在初始"tight mode"下并行下载，减少等待布局完成的时间。

在 DevTools Network 面板中，开启"Big request rows"设置或悬停可查看优先级的初始值和最终值变化。

## Preload Scanner（预加载扫描器）

### 工作原理

浏览器有两个 HTML 解析器协同工作：

1. **主 HTML 解析器（Primary Parser）**：逐 token 解析 HTML，构建 DOM。遇到阻塞资源（如 `<link rel="stylesheet">` 或无 `async`/`defer` 的 `<script>`）时会暂停。
2. **Preload Scanner（预加载扫描器）**：在主解析器被阻塞时，**并行扫描原始 HTML 文本**，提前发现 `<img>`、`<link>`、`<script>` 等资源标签并发起请求。

```
HTML 文档
  ├── 主解析器 → 遇到 <link rel="stylesheet"> → 阻塞（等待 CSS 下载+解析）
  └── Preload Scanner → 继续扫描 → 发现 <img src="..."> → 提前发起请求
```

### 关键限制

Preload Scanner **只扫描 HTML 标记**，不扫描其他资源类型：

- **不会扫描 CSS 文件**：CSS 中的 `background-image`、`@font-face` 等资源要等 CSSOM 构建时才会被发现
- **不会扫描 JavaScript**：JS 动态创建的资源标签对 Preload Scanner 不可见
- **不识别非标准属性**：如 `data-src`（常见于 JS 懒加载方案）不会被识别为资源 URL

## Fetch Priority API（`fetchpriority` 属性）

Fetch Priority API 原名 Priority Hints，标准化后更名为 Fetch Priority。它允许开发者通过 `fetchpriority` HTML 属性或 Fetch API 的 `priority` 选项，向浏览器提示资源的相对优先级。

### 浏览器支持

| Chrome | Edge | Firefox | Safari |
|---|---|---|---|
| 102+ | 102+ | 132+ | 17.2+ |

### HTML 属性用法

`fetchpriority` 属性可用于 `<link>`、`<img>`、`<script>` 元素，接受三个值：

| 值 | 含义 |
|---|---|
| `high` | 提示浏览器提高该资源的优先级 |
| `low` | 提示浏览器降低该资源的优先级 |
| `auto` | 默认值，由浏览器自行决定 |

```html
<!-- 提升 LCP 图片优先级 -->
<img src="lcp-image.jpg" fetchpriority="high" alt="Hero image">

<!-- 降低非关键图片优先级 -->
<img src="carousel-2.jpg" fetchpriority="low" alt="Carousel slide 2">

<!-- 降低预加载脚本的优先级 -->
<link rel="preload" href="/js/non-critical.js" as="script" fetchpriority="low">

<!-- 提升 async 脚本优先级 -->
<script src="critical-async.js" async fetchpriority="high"></script>
```

### Fetch API 用法

```javascript
// 默认高优先级的 fetch
let userData = await fetch('/api/user');

// 显式降低优先级的后台数据请求
let suggestedContent = await fetch('/api/suggestions', { priority: 'low' });
```

### `fetchpriority` 对各资源类型的实际影响

`fetchpriority` 设置的是**相对优先级**（raise/lower），而非绝对优先级：

| 资源类型 | 默认 | `fetchpriority="high"` | `fetchpriority="low"` |
|---|---|---|---|
| CSS（早期） | Highest | Highest（已最高，无法再提升） | High |
| CSS（晚期） | High | Highest | Medium |
| Script（早期） | Highest | Highest | High |
| Script（async/defer） | Low | High | Lowest |
| Font | Highest | Highest | High |
| Image（视口内） | High | Highest | Medium |
| Image（普通） | Low | Medium | Lowest |
| XHR/Fetch | High | Highest | Low |

> **关键理解**：`fetchpriority="high"` 不等于"设为 High 优先级"，而是"在默认基础上提升一级"。如果资源已经是最高优先级，则保持不变。

## 典型使用场景

### 场景 1：提升 LCP 图片加载速度

**问题**：LCP 图片（通常是首屏大图）初始优先级为 `Low`，需要等待布局完成后才被提升为 `High`，造成不必要的延迟。

**方案**：

```html
<img src="lcp-image.jpg" fetchpriority="high" alt="Hero image">
```

**效果**：Google Flights 实验中，LCP 从 2.6s 降低到 1.9s。

对于 **CSS 背景图片**作为 LCP 的情况，需要结合 `preload` 和 `fetchpriority`：

```html
<link rel="preload" as="image" href="lcp-bg.jpg" fetchpriority="high">
```

### 场景 2：轮播图优先级优化

**问题**：轮播图中所有图片都在视口附近，浏览器可能将它们都提升为高优先级，导致带宽竞争。

**方案**：

```html
<ul class="carousel">
  <img src="carousel-1.jpg" fetchpriority="high" alt="First slide">
  <img src="carousel-2.jpg" fetchpriority="low" alt="Second slide">
  <img src="carousel-3.jpg" fetchpriority="low" alt="Third slide">
  <img src="carousel-4.jpg" fetchpriority="low" alt="Fourth slide">
</ul>
```

即使图片 2-4 在视口外，如果它们"足够接近"视口，浏览器仍可能提升其优先级。`fetchpriority="low"` 可以明确告诉浏览器降低这些图片的优先级。

### 场景 3：Async 脚本优先级提升

**问题**：`async` 脚本默认 `Low` 优先级，但对交互至关重要的 async 脚本需要更快加载。

**方案**：

```html
<script src="critical-async.js" async fetchpriority="high"></script>
```

这比使用 `<link rel="preload">` 作为 hack 方案语义更清晰。

### 场景 4：降低非关键数据请求优先级

**问题**：`fetch()` 默认 `High` 优先级，但后台数据请求不应与用户交互相关的请求竞争带宽。

**方案**：

```javascript
// 用户认证数据（保持默认高优先级）
let auth = await fetch('/api/auth');

// 后台推荐内容（降低优先级）
let recommendations = await fetch('/api/recommendations', { priority: 'low' });
```

### 场景 5：降低预加载资源的竞争

**问题**：`preload` 是强制下载（非提示），如果预加载了非关键资源，可能与关键资源竞争带宽。

**方案**：

```html
<!-- 关键脚本正常预加载 -->
<link rel="preload" as="script" href="critical.js">

<!-- 非关键脚本降低优先级预加载 -->
<link rel="preload" as="script" href="non-critical.js" fetchpriority="low">

<!-- 非关键 CSS 预加载且不阻塞渲染 -->
<link rel="preload" as="style" href="theme.css" fetchpriority="low"
      onload="this.rel='stylesheet'">
```

## Fetch Priority 与 Preload 的区别

这是两个互补但不同的机制：

| 维度 | Preload（`rel="preload"`） | Fetch Priority（`fetchpriority`） |
|---|---|---|
| 本质 | **强制下载**（mandatory fetch） | **优先级提示**（hint） |
| 解决的问题 | **何时发现**资源（discoverability） | **以什么优先级**下载资源 |
| 是否可省略 | 否，浏览器必须下载 | 是，浏览器可忽略提示 |
| 与 LCP 的关系 | 让浏览器提前发现 LCP 资源 | 让 LCP 资源以更高优先级下载 |

**组合使用示例**（CSS 背景 LCP 图片）：

```html
<!-- preload 解决"提前发现"，fetchpriority 解决"高优先级下载" -->
<link rel="preload" as="image" href="lcp-bg.jpg" fetchpriority="high">
```

## 与 HTTP/2 和 HTTP/3 优先级的关系

- Fetch Priority 在**浏览器内部**和**支持优先级的协议**（HTTP/2、HTTP/3）中都会生效。
- 即使 CDN 不实现 HTTP/2 优先级调度，浏览器内部的优先级调度仍然有效（例如低优先级资源在浏览器处理关键 `<head>` 项时被延迟发起请求）。
- CDN 对 HTTP/2/HTTP/3 优先级的实现不统一，这使得在不同 CDN 环境下测试 Fetch Priority 效果变得复杂。
- 在带宽受限环境（HTTP/1.x 或低速 HTTP/2/3）下，优先级的效果最为明显。

## 在 DevTools 中查看优先级

1. 打开 Chrome DevTools → Network 面板
2. 右键点击列标题 → 勾选 **Priority** 列
3. 开启 **Big request rows** 设置可查看优先级的初始值和最终值
4. 悬停资源行也可在 tooltip 中查看优先级变化

## 限制与注意事项

1. **`fetchpriority` 是提示而非指令**：浏览器会参考开发者的偏好，但最终决策权在浏览器。浏览器可能根据自身资源优先级偏好解决冲突。
2. **不要过度使用**：如果所有资源都设为高优先级，等于没有优先级。优先关注对 Core Web Vitals 影响最大的资源。
3. **不要与 preload 混淆**：preload 是强制下载，fetchpriority 是优先级提示。两者解决的问题不同。
4. **CDN 兼容性**：CDN 对 HTTP/2 优先级的实现不一致，可能影响实际效果。
5. **测量困难**：优先级变更的效果通常比 preload 更难观察和量化，需要结合 WebPageTest 等工具进行对比测试。
6. **CSS media 不匹配的特殊情况**：即使设置 `fetchpriority="high"`，media 不匹配的 CSS（如 `media="print"` 在屏幕设备上）仍然不会被 Preload Scanner 发现，只能在主解析器到达时才处理。
7. **避免 JS 注入脚本对抗 Preload Scanner**：通过 `document.createElement('script')` 注入的脚本不会被 Preload Scanner 发现，应直接在 HTML 中使用 `<script>` 标签。
8. **避免对首屏图片使用 JS 懒加载**：使用 `data-src` 等自定义属性的懒加载方案会阻止 Preload Scanner 发现图片，应让首屏图片使用原生 `src` 属性。
9. **避免过度内联资源**：内联过多 CSS/字体/JS 会延迟 Preload Scanner 发现后续资源，因为下载更大的 HTML 文档需要更长时间。
10. **客户端渲染（CSR）的代价**：完全由 JS 渲染的页面，所有资源对 Preload Scanner 不可见，应考虑 SSR 或静态生成。

## 最佳实践总结

1. **优先优化 LCP 资源**：对 LCP 图片使用 `fetchpriority="high"`，对 CSS 背景 LCP 图片使用 `preload + fetchpriority="high"`。
2. **降低非关键资源优先级**：轮播图非首帧图片、后台数据请求、非关键脚本使用 `fetchpriority="low"`。
3. **让 Preload Scanner 工作**：避免 JS 注入关键脚本、避免对首屏资源使用 JS 懒加载、避免过度内联。
4. **preload 要节制**：只预加载浏览器无法提前发现的关键资源（字体、CSS 背景图、JS 动态导入的模块）。
5. **测量验证**：使用 WebPageTest 对比测试，关注真实用户数据（RUM）而非仅依赖实验室数据。
6. **HTTP 头 preload 优先级最高**：通过 `Link` HTTP 头发送的 preload 会出现在加载顺序的最前面。
