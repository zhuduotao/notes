死锁（Deadlock）与活锁（Livelock）是并发与分布式系统中的两类典型阻塞问题，均会导致系统无法正常推进，但成因与表现形式截然不同。

---

## 死锁（Deadlock）

### 定义

死锁是指一组进程（或线程）相互等待对方释放资源，且都不主动释放自己已持有的资源，导致所有参与者永久阻塞的状态。

经典定义源自 Coffman 等人（1971）提出的四个必要条件，必须同时满足才会发生死锁：

| 条件 | 含义 |
|------|------|
| 互斥（Mutual Exclusion） | 资源不能被多个进程同时占用 |
| 持有并等待（Hold and Wait） | 进程持有部分资源，同时等待其他资源 |
| 不可抢占（No Preemption） | 已分配的资源不能被强制回收 |
| 循环等待（Circular Wait） | 存在进程等待环，A 等 B，B 等 C，...，C 等 A |

### 常见场景

1. **数据库事务**：两个事务分别持有表 A、表 B 的锁，并尝试获取对方的锁
2. **线程编程**：两个线程以不同顺序获取多把锁（如 Java 的 `synchronized` 或 `ReentrantLock`）
3. **分布式系统**：节点间资源协调不当，形成跨节点等待环

### 处理策略

| 策略 | 思路 | 适用场景 |
|------|------|----------|
| 预防（Prevention） | 破坏 Coffman 条件之一（如统一锁顺序） | 可控环境，设计阶段 |
| 避免（Avoidance） | 动态检查资源分配是否安全（如银行家算法） | 资源需求可预测 |
| 检测与恢复（Detection & Recovery） | 定期检测死锁，通过终止进程或抢占资源解除 | 允许短暂死锁，有恢复机制 |
| 忽略（Ignore） | 假设死锁概率极低，不做处理 | 大多数操作系统（如 Linux、Windows） |

### 示例（Java）

```java
// 锁顺序不当导致死锁
public void transfer(Account from, Account to, int amount) {
    synchronized (from) {  // 先锁 from
        synchronized (to) { // 再锁 to
            from.debit(amount);
            to.credit(amount);
        }
    }
}
// 若线程 A: transfer(a1, a2), 线程 B: transfer(a2, a1) → 死锁
```

正确做法：按固定顺序（如账户 ID）获取锁，或使用全局锁。

## 活锁（Livelock）

### 定义

活锁是指进程持续改变状态、响应事件，但始终无法完成有效工作，看似"活跃"实则无法推进。与死锁的区别在于：进程并未静止等待，而是不断执行无效动作。

典型比喻：两人走廊相遇，均礼貌地向同一侧避让，反复碰撞，永远无法通过。

### 常见场景

1. **分布式共识**：多个 Proposer 交替提升提案编号，导致提案反复被抢占（如 Paxos 活锁）
2. **网络重传**：两个节点同时重传冲突数据包，互相触发对方重传
3. **锁竞争退避**：多个线程同时检测冲突并退避，退避策略同步导致反复冲突

### 防止方法

- **引入随机性**：退避时间随机化，避免同步重试（如以太网 CSMA/CD）
- **优先级机制**：赋予部分参与者更高优先级，打破对称性（如 Paxos 中的 Leader）
- **超时限制**：超过一定时间后改变策略或放弃
- **确定性仲裁**：引入外部仲裁者或领导者

### 示例（Paxos 活锁）

Paxos 协议中，若多个 Proposer 同时提出提案，可能交替提升编号抢占对方提案，导致长期无法达成共识。解决方案：选举单一 Leader 提案，或对编号提升速率设限。

## 死锁与活锁对比

| 维度 | 死锁 | 活锁 |
|------|------|------|
| 进程状态 | 阻塞等待 | 活跃执行（无效） |
| CPU 占用 | 低（通常为 0） | 高（持续执行） |
| 检测难度 | 相对容易（等待环可检测） | 较难（需识别无效进展） |
| 常见成因 | 循环等待资源 | 退避策略同步、对称竞争 |
| 典型解决 | 破坏循环等待 | 引入随机性或优先级 |

## 相关概念

- **饥饿（Starvation）**：进程长期无法获取资源，被其他进程持续抢占；与死锁不同，系统中其他进程仍可推进
- **竞态条件（Race Condition）**：执行结果依赖执行顺序，可能引发数据错误
- **锁粒度**：锁范围越大，死锁概率越高，但并发度越低
- **锁层级**：如 VFS 文件系统的锁层级规则，防止死锁

---

## 注意事项

- 死锁四个条件必须同时满足，破坏任一即可预防
- 活锁并非"活的死锁"，而是另一种阻塞形式
- 实际系统中，死锁检测成本较高，多数采用预防策略
- 分布式系统需额外考虑网络延迟、节点故障等因素
