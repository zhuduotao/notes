## 定义

**缓存一致性（Cache Coherence）** 是指在存在多个缓存副本的情况下，确保所有缓存副本与主存（或其他权威数据源）的数据保持一致的机制或属性。

这个概念在不同层面有不同含义：

| 层面 | 语境 | 核心问题 |
|------|------|----------|
| **硬件层** | 多核 CPU 的 L1/L2 缓存 | 多个核心对同一内存地址的读写视图是否一致 |
| **I/O 层** | DMA 与 CPU 缓存 | 设备写入内存后，CPU 是否能读到最新数据 |
| **应用层** | 分布式缓存（Redis、本地缓存 + DB） | 缓存与数据库、多节点缓存之间数据是否一致 |

## 为什么重要

### 硬件层

现代多核 CPU 每个核心有独立的 L1/L2 缓存。若没有一致性机制：

- 核心 A 写入 `x = 1`，值仅存在于 A 的缓存中
- 核心 B 读取 `x`，可能读到旧值（B 缓存中的副本或主存中的旧值）
- 程序行为违反程序员直觉，并发正确性无法保证

缓存一致性是**硬件锁机制（CAS、LL/SC）的底层基础**——没有一致性协议，多核原子操作无法实现。

### I/O 层

DMA 设备直接写入内存，绕过 CPU 缓存：

- 设备写入数据后，CPU 缓存中可能仍持有旧值
- CPU 读到过期数据，导致协议解析错误、数据损坏

### 应用层

分布式系统中，缓存（Redis、本地缓存）与数据库分离：

- 写数据库后缓存未更新 → 读到旧数据
- 缓存更新后数据库写入失败 → 数据不一致
- 多节点缓存之间副本不同步

## 硬件层缓存一致性

### MESI 协议

MESI 是最广泛使用的写回（Write-back）缓存一致性协议，Intel、AMD x86 处理器采用 [^mesi]。

每个缓存行的四种状态：

| 状态 | 含义 | 可执行操作 |
|------|------|------------|
| **Modified (M)** | 缓存行已修改，仅存在于当前缓存，与主存不一致 | 可自由读写；其他核心请求时需回写主存 |
| **Exclusive (E)** | 缓存行未修改，仅存在于当前缓存，与主存一致 | 可自由读写；写入后转为 M 状态 |
| **Shared (S)** | 缓存行未修改，可能存在于多个缓存中 | 可读；写入需发送失效信号 |
| **Invalid (I)** | 缓存行无效 | 无任何操作权限 |

**状态转换关键场景**：

1. **核心 A 读入数据，无其他缓存持有** → A 进入 E 状态（独占）
2. **核心 B 也读入同一地址** → A、B 都进入 S 状态（共享）
3. **核心 A 写入数据** → A 发送失效信号，A 进入 M 状态，B 进入 I 状态
4. **核心 B 读取被 A 修改的地址** → A 回写主存，A 进入 S 状态，B 进入 S 状态

### MOESI 协议

AMD 采用 MOESI 协议，增加 **Owned (O)** 状态 [^mesi]：

- **Owned (O)**：缓存行已修改，但其他缓存可能持有 S 状态副本；O 状态缓存负责向其他缓存提供数据，无需立即回写主存

优势：减少回写主存的次数，降低总线带宽消耗。

### MESIF 协议

Intel 多路系统使用 MESIF 协议，增加 **Forward (F)** 状态 [^mesi]：

- **Forward (F)**：在多个 S 状态缓存中指定一个作为"转发者"，响应其他核心的读请求

优势：避免多个 S 状态缓存同时响应读请求造成的总线冲突。

### 硬件实现机制

**Bus Snooping（总线嗅探）**：缓存控制器监听总线上的读写请求，根据协议规则更新本地缓存状态。

**关键总线消息**：

| 消息 | 作用 |
|------|------|
| `BusRd` | 请求读取共享副本 |
| `BusRdX` | 请求读取独占副本（意图写入） |
| `BusUpgr` | 请求升级为 M 状态（已有 S 状态时） |
| `Flush` | 回写主存 |
| `FlushOpt` | 将数据直接发给请求者（无需经过主存） |

### 内存序与缓存一致性

缓存一致性解决的是**单地址的可见性问题**；内存序解决的是**多地址操作的可见顺序问题**。两者配合才能保证并发程序正确性 [^intel-sdm]。

| 架构 | 内存序模型 | 缓存一致性协议 |
|------|-----------|---------------|
| x86 (Intel/AMD) | TSO（强） | MESI / MESIF |
| ARM | Weak（弱） | 实现定义（通常 MESI 变体） |
| RISC-V | Weak（弱） | 实现定义 |
| PowerPC | Weak（弱） | 实现定义 |

## I/O 层缓存一致性

### 问题场景

DMA 操作绕过 CPU 缓存，直接访问主存 [^dma]：

| 场景 | 问题 |
|------|------|
| **DMA 写入内存** | CPU 缓存中持有旧值，读回过期数据 |
| **DMA 读取内存** | CPU 已修改数据但未写回（Write-back Cache），设备读到旧值 |

### 解决方案

| 方案 | 机制 | 适用场景 |
|------|------|----------|
| **硬件方案：Bus Snooping** | DMA 写入时缓存控制器自动失效/刷新对应缓存行 | 缓存一致性系统（Cache-coherent systems） |
| **软件方案：OS 管理** | 传输前 flush 缓存，传输后 invalidate 缓存 | 非一致性系统（Non-coherent systems） |
| **使用一致性内存** | `dma_alloc_coherent()` 分配不会被缓存的内存 | Linux 内核驱动 |

**Linux 内核 API**（非一致性系统）：

```c
// DMA 前刷新 CPU 缓存到内存
dma_sync_single_for_device(dev, dma_addr, size, DMA_TO_DEVICE);

// DMA 后失效 CPU 缓存（使 CPU 读内存新值）
dma_sync_single_for_cpu(dev, dma_addr, size, DMA_FROM_DEVICE);
```

### 硬件增强

**DDIO（Data Direct I/O）**：Intel Xeon E5 引入，允许 DMA 数据驻留在 L3 Cache 中 [^dma]，避免 CPU 从主存读取 I/O 数据的开销。

**IOMMU**：提供 DMA 地址转换，同时可配置 DMA 缓存一致性策略 [^dma]。

## 应用层缓存一致性

分布式系统中，缓存一致性是**架构设计问题**，而非硬件自动解决的问题。

### 缓存更新策略

| 策略 | 写操作流程 | 读操作流程 | 特点 |
|------|-----------|-----------|------|
| **Cache-Aside** | 先写 DB，再删缓存 | 先读缓存，miss 则读 DB 并写缓存 | 最常用；存在短暂不一致窗口 |
| **Write-Through** | 先写缓存（同步写 DB），缓存更新完成才返回 | 同 Cache-Aside | 数据强一致；写入延迟高 |
| **Write-Back (Write-Behind)** | 先写缓存，异步写 DB | 同 Cache-Aside | 写入延迟低；DB 可能滞后，宕机丢数据 |
| **Write-Around** | 直接写 DB，不更新缓存 | 同 Cache-Aside | 适合写多读少；读时才加载缓存 |

### Cache-Aside 的并发问题

**问题 1：先删缓存，再写 DB → 读请求回填旧数据**

```
时间线：
1. 线程 A 删除缓存
2. 线程 B 读缓存 miss，读 DB（旧值）
3. 线程 B 写入缓存（旧值）
4. 线程 A 写入 DB（新值）

结果：缓存中是旧值，DB 中是新值
```

**解决方案**：采用"先写 DB，再删缓存"顺序，缩短不一致窗口。

**问题 2：写 DB + 删缓存的原子性**

若 DB 写入成功但删缓存失败 → 缓存中仍是旧值。

**解决方案**：

- **重试机制**：删缓存失败时异步重试
- **延迟双删**：写 DB 后删缓存，延迟 N 秒后再删一次（防止并发回填）
- **Binlog 订阅**：通过数据库 binlog 异步删除缓存（解耦写入与删除）

### 分布式缓存的一致性

多节点缓存（如 Redis Cluster）之间的一致性 [^redis]：

| 问题 | 解决方案 |
|------|----------|
| **主从同步延迟** | 主节点写入后，从节点可能滞后；读从节点可能读到旧值 | 读主节点；或接受最终一致 |
| **分布式锁竞态** | 多节点同时获取锁可能导致不一致 | 使用 Redlock 算法（多节点加锁，多数成功才算成功） |
| **缓存分片** | 不同 key 分布在不同节点，跨 key 操作无原子性保证 | 使用分布式事务或接受最终一致 |

**Redlock 算法要点** [^redis]：

1. 获取当前时间戳
2. 依次向 N 个 Redis 节点请求加锁
3. 计算加锁耗时，若在有效时间内且多数节点成功，则加锁成功
4. 否则释放所有节点的锁

### 一致性级别

| 级别 | 定义 | 实现代价 |
|------|------|----------|
| **强一致** | 读操作总是返回最新写入值 | 高延迟、低可用（如分布式锁 + 写后读主） |
| **最终一致** | 系统保证最终（无新写入时）所有副本一致 | 低延迟、高可用（异步同步） |
| **弱一致** | 不保证任何一致性，读可能返回任意值 | 最低延迟（如本地缓存无同步） |

### CAP 理论

CAP 定理（Brewer's Conjecture）由 Eric Brewer 于 2000 年提出，指出分布式系统在网络分区发生时，最多只能同时满足三项中的两项 [^cap]：

| 属性 | 定义 | 实现代价 |
|------|------|----------|
| **C (Consistency)** | 所有节点在同一时刻看到的数据相同（线性一致性） | 高延迟、需同步协调 |
| **A (Availability)** | 每个请求都能在有限时间内获得响应 | 允许返回过期数据 |
| **P (Partition Tolerance)** | 网络分区发生时系统仍能继续运行 | 必须选择牺牲 C 或 A |

**核心结论**：网络分区在分布式系统中不可避免，因此实际选择是 **CP**（牺牲可用性保一致）或 **AP**（牺牲一致性保可用）。不存在真正意义上的"CA 系统"——单机系统没有分区问题。

**典型系统的 CAP 选择**：

| 系统 | 选择 | 说明 |
|------|------|------|
| ZooKeeper | CP | 保证强一致，分区时可能拒绝服务 |
| Redis Cluster | AP | 主从异步复制，允许短暂不一致 |
| Cassandra | AP | 可调一致性，默认追求可用性 |
| HBase | CP | 强一致，依赖 HDFS |
| DynamoDB | 可调 | 支持配置一致性级别 |

### 保证读取最新数据的策略

#### 1. Read-Your-Writes（读己之写）

保证用户写入后，后续读取能看到自己的写入 [^session-guarantees]。

**实现方式**：

- **会话绑定**：同一用户的读写请求路由到同一节点（如 sticky session）
- **版本追踪**：客户端记录最后一次写入的版本号/时间戳，读取时要求版本 ≥ 该值
- **写后读主节点**：写入主节点后，读取也走主节点（牺牲读性能）

#### 2. Monotonic Reads（单调读）

保证用户不会看到数据"回退"——若读了 v2，后续不会读到 v1 [^session-guarantees]。

**实现方式**：

- **会话绑定**：同上
- **读版本递增**：客户端记录已读的最大版本，后续读取要求版本 ≥ 该值

#### 3. Quorum Reads（法定人数读）

基于分布式共识，要求读取足够多的副本以保证看到最新写入 [^quorum]。

**规则**：

- 写入需要 `W` 个副本确认
- 读取需要查询 `R` 个副本
- 保证最新写入的条件：`W + R > N`（N 为副本总数）

**示例**：N=3, W=2, R=2 → 写入需 2 副本确认，读取需查 2 副本，交集至少 1 个副本有最新值。

**权衡**：

- `W + R > N`：强一致，但延迟高
- `W + R ≤ N`：可能读到旧值，但延迟低、可用性高

#### 4. 版本控制方案

**版本号/时间戳**：

- 每次写入附带递增版本号或时间戳
- 读取时返回版本号，客户端可判断新旧
- 多副本返回不同版本时，取最大版本

**Vector Clock（向量时钟）** [^vector-clock]：

- 每个节点维护自己的计数器
- 写入时增加本节点计数器
- 读取时比较向量时钟判断因果关系
- 可检测并发写入冲突，需客户端或应用层解决冲突

#### 5. 具体系统实践建议

**Redis Cluster**：

| 场景 | 建议 |
|------|------|
| 强一致需求 | 读写都走主节点；或使用 `WAIT` 命令等待副本同步 |
| 最终一致可接受 | 读从节点，接受短暂不一致 |
| 分布式锁 | 使用 Redlock 算法，多节点加锁 |

**本地缓存 + Redis + DB 三层架构**：

| 策略 | 实现 |
|------|------|
| **写后失效** | 写 DB → 删 Redis → 删本地缓存（广播失效消息） |
| **写后更新版本** | 写 DB → 更新 Redis 版本 → 本地缓存比对版本失效 |
| **定时刷新** | 本地缓存设置 TTL，定期从 Redis 拉取最新值 |
| **订阅变更** | 本地缓存订阅 Redis Pub/Sub 或消息队列，收到变更通知即失效 |

**注意事项**：

- 本地缓存难以保证实时一致，通常只能做到最终一致
- 广播失效消息需考虑消息丢失、延迟、重复失效等场景
- 高并发场景优先考虑"失效"而非"更新"缓存，避免并发更新写入不一致值

## 常见误区

1. **"缓存一致性是硬件自动解决的问题"**

   硬件层（MESI）解决的是多核 CPU 缓存一致性；应用层（分布式缓存）需要架构设计解决。

2. **"Cache-Aside 策略保证强一致"**

   Cache-Aside 只能缩短不一致窗口，无法保证强一致。并发场景下仍可能回填旧数据。

3. **"删缓存比更新缓存好"**

   删除缓存可避免并发更新写入不一致值；但并发回填问题仍需额外处理（延迟双删、Binlog 订阅）。

4. **"Write-Through 比 Cache-Aside 更好"**

   Write-Through 保证强一致但写入延迟高；Cache-Aside 最终一致但写入延迟低。需根据业务需求选择。

5. **"MESI 协议中 S 状态可以写入"**

   错误。S 状态写入需发送失效信号，将其他缓存副本置为 I 状态后才能写入。

6. **"缓存一致性解决了所有并发问题"**

   缓存一致性解决单地址可见性；多地址操作顺序需内存序（Memory Order）配合。
